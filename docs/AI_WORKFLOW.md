# BobFull AI 협업 워크플로우

## 1. 목적

최초 1회 에이전트를 온보딩한 뒤, 팀원은 긴 프롬프트 대신 짧은 명령으로 작업한다.

구현 에이전트:

```text
Issue #번호 구현하라
```

리뷰 에이전트:

```text
PR #번호 1차 리뷰하라
```

AI는 현재 Issue·PR 상태를 읽고 다음 단계만 수행한다. Human은 이해도 질문에 답하고 최종 코드를 확인한 뒤 Merge한다.

## 2. 전체 흐름

```text
1. 팀원별 AI 에이전트 최초 온보딩
2. 구현 AI가 Issue·문서·코드 분석과 Human 질문 작성
3. 담당자가 Issue의 Human 이해도 답변 작성
4. 구현 AI가 답변 검증·AI 보완 설명·최종 Issue 재작성
5. Human이 최종 Issue를 읽은 뒤 다시 `Issue #번호 구현하라` 입력
6. 구현 AI가 구현·테스트·Commit·Push·Draft PR 생성
7. Draft PR 이후 아래 두 작업은 순서와 관계없이 진행 가능
   - 담당자가 PR의 Human 이해도 답변 작성
   - 다른 팀원의 리뷰 AI가 현재 Head 기준 1차 리뷰 수행
8. Human 답변과 1차 리뷰가 모두 등록되면 구현 AI가 범위 안 지적 수정·재검증·Push
9. 수정된 최신 Head를 리뷰 AI가 재검토
10. Human이 최종 코드 확인 후 Merge
```

Issue·PR 양식에는 Human 체크리스트, 승인자, 확인 범위 입력란을 두지 않는다. Human이 직접 작성하는 항목은 이해도 답변뿐이다.

1차 AI 리뷰는 PR 완료 판정 단계가 아니다. 구현 중이거나 테스트·Human 답변이 미완료인 PR도 현재 Head의 결함과 미검증 위험을 찾기 위해 리뷰할 수 있다.

## 3. Issue 단계

### 3.1 Issue 분석

구현 AI는 Issue, 확정 문서, 관련 코드와 테스트를 확인한다.

- 목적과 범위·제외 범위
- 확정 정책과 미결정 사항
- 정상·실패·경계 흐름
- API·DB·상태·권한·트랜잭션
- 선행 작업과 도메인 의존성
- 완료 조건과 검증 계획
- 문서·Issue·코드 충돌

Human 결정이 필요하면 Issue의 `Human 이해도`에 구체적인 질문을 작성하고 상태를 `HUMAN_ANSWER_REQUIRED`로 남긴 뒤 중단한다.

### 3.2 Human 답변

담당자는 Issue의 질문에 직접 답한다. 별도의 체크·승인 문구는 작성하지 않는다.

### 3.3 AI 검증과 최종 Issue 재작성

같은 명령을 다시 받으면 구현 AI는 Human 답변을 문서·코드와 대조한다.

- 답변이 맞으면 `AI 보완 설명`으로 필요한 맥락을 추가한다.
- 답변이 부족하거나 잘못됐으면 `보완 필요`와 근거를 남긴다.
- 답변을 반영해 범위·정책·흐름·완료 조건·검증 계획을 최종 계약으로 재작성한다.
- 충돌이나 미결정 사항이 남으면 `HOLD`로 중단한다.
- 구현 가능한 계약이면 `AI_FINALIZED`로 남긴다.

Human은 최종 Issue를 읽고 수정이 필요하면 AI에게 요청한다. 문제가 없으면 다시 `Issue #번호 구현하라`라고 입력한다. 이 명령이 구현 지시가 된다.

## 4. 구현과 Draft PR

Issue가 `AI_FINALIZED`인 상태에서 같은 명령을 받으면 구현 AI는 다음을 수행한다.

```text
기능 브랜치 확인 또는 생성
→ 최소 변경 계획
→ 구현
→ 테스트·직접 검증
→ 실제 Diff 자체 검토
→ PR Human 이해도 질문 작성
→ Commit·Push
→ develop 대상 Draft PR 생성
```

필수 원칙:

- 최종 Issue 계약 범위만 구현한다.
- 실행하지 않은 테스트를 PASS로 기록하지 않는다.
- build 실패를 성공으로 표현하지 않는다.
- 정책·API·DB 재결정이 필요하면 중단한다.
- PR에는 실제 변경·실행 결과·미검증 위험만 기록한다.

## 5. PR Human 답변

Draft PR 생성 후 담당자는 실제 Diff와 테스트 결과를 읽고 PR의 Human 이해도 질문에 답한다.

Human이 작성할 항목은 답변뿐이다. 체크리스트, 승인자, 확인 범위는 작성하지 않는다.

Human 답변은 리뷰 AI가 담당자 이해도를 확인하고 Merge 전 위험을 정리하는 데 필요하다. 다만 답변이 아직 없더라도 1차 AI 리뷰는 진행할 수 있으며, 리뷰 AI는 답변 상태를 `미작성`으로 기록한다.

## 6. AI 1차 리뷰

PR 작성자가 아닌 다른 팀원의 AI 에이전트가 `PR #번호 1차 리뷰하라` 명령으로 리뷰한다.

### 최소 시작 조건

- 최신 Head Commit SHA
- 최신 Head의 실제 Diff
- Issue 또는 PR 본문에서 확인할 수 있는 작업 범위

위 세 항목을 확인할 수 있으면 PR 상태와 관계없이 1차 리뷰를 수행한다.

가능한 범위에서 함께 확인한다.

- 최종 Issue 계약
- PR 본문
- 관련 코드와 테스트
- 테스트·build·직접 검증·CI 증거
- 미검증 범위와 남은 위험
- Human 이해도 답변

다음 상태는 리뷰 중단 사유가 아니다.

- Draft 또는 구현 중 PR
- Issue 계약 미완료
- 테스트·build 실패
- 테스트·직접 검증·CI 미실행
- `FAIL`, `HOLD`, `NOT_RUN` 항목 존재
- Human 답변 미작성·불완전·`모름`
- PR 설명과 실제 Diff 불일치

리뷰 AI는 이러한 미완료 사항을 결함, 계약 충돌, 확인 필요 또는 미검증 위험으로 기록한다. 최신 Head SHA나 실제 Diff가 없거나 작업 범위를 전혀 확인할 수 없을 때만 `1차 리뷰 준비 미완료`로 보고한다.

리뷰 AI는 결함과 위험을 검토하고 Human 답변이 있으면 최신 코드와 대조한다.

- 정확하면 `일치`
- 부족하면 `AI 보완 설명`
- 잘못됐으면 `보완 필요`와 코드 근거
- 답변이 없으면 `미작성`으로 기록하고 코드 리뷰는 계속함

1차 리뷰 완료는 구현 완료, 테스트 통과 또는 Merge 가능을 의미하지 않는다.

리뷰 AI는 코드·PR을 수정하거나 Approve·Merge하지 않는다.

## 7. 리뷰 수정

구현 AI가 다시 `Issue #번호 구현하라` 명령을 받으면 연결된 PR과 1차 리뷰를 확인한다.

자동 수정 가능한 항목:

- Issue 계약 안의 명확한 기능 오류
- 예외 처리·검증·테스트 누락
- 문서·PR 설명 불일치
- 정책 재결정이 필요 없는 BLOCKER·MAJOR·MINOR

자동 수정하지 않는 항목:

- QUESTION
- 범위 확장
- 정책·API·DB·상태·권한·트랜잭션 재결정
- 다른 담당자 계약 변경
- 새로운 라이브러리·인프라 도입

Human 이해도 답변이 아직 없으면 리뷰 자체는 유효하지만, 구현 AI는 답변과 Human 결정이 필요한 항목을 기다린다. 명확한 범위 안 코드 지적은 Human 결정이 필요한 항목과 분리해 보고한다.

수정 후 테스트·build·직접 검증을 다시 수행하고 Commit·Push와 PR 본문을 갱신한다.

## 8. 최신 Head 재검토

수정 Commit이 올라오면 리뷰 AI는 이전 결과를 그대로 재사용하지 않는다.

- 현재 Head SHA와 수정 Diff 확인
- 기존 지적 해결 여부 확인
- 영향받은 테스트·build·직접 검증·CI 확인
- 새 결함과 PR 설명 불일치 확인
- Human 답변이 추가·수정됐으면 최신 코드와 재대조

해결되지 않은 항목과 미검증 위험을 최신 상태로 다시 기록한다.

## 9. 최종 Human 확인과 Merge

구현 AI가 `FINAL_HUMAN_REVIEW`로 보고하면 Human이 실제 코드, 리뷰 수정 결과, 테스트·CI와 남은 위험을 확인한다.

다음을 충족하지 않으면 Merge 준비 완료로 판단하지 않는다.

- 담당자의 Human 이해도 답변 확인
- 필수 테스트·build·직접 검증 결과 확인
- 최신 Head 기준 AI 재검토
- 해결되지 않은 BLOCKER 없음
- GitHub Rules의 Human Approve 조건 충족

PR 작성자가 Merge하며 AI는 Approve와 Merge를 수행하지 않는다.

## 10. 실행 재개 원칙

AI 에이전트는 Human이 Issue·PR을 수정한 뒤 스스로 다시 실행되지 않는다.

각 Human 답변 뒤에는 같은 짧은 명령을 다시 사용한다.

```text
Issue #번호 구현하라
```

구현 AI는 처음부터 반복하지 않고 실제 Issue·PR 상태를 읽어 다음 단계부터 재개한다.
