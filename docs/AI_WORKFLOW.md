# BobFull AI 협업 워크플로우

## 1. 목적

최초 1회 에이전트를 온보딩한 뒤, 팀원은 긴 프롬프트 대신 짧은 명령으로 작업을 진행한다.

구현 에이전트:

```text
Issue #번호 구현하라
```

리뷰 에이전트:

```text
PR #번호 1차 리뷰하라
```

AI는 현재 Issue·PR 상태를 읽고 허용된 다음 단계만 수행한다. Human은 Issue·PR에 직접 답변하고 최종 승인과 Merge를 책임진다.

## 2. 전체 흐름

```text
1. 팀원별 AI 에이전트 최초 온보딩
2. 구현 AI가 Issue·문서·코드 분석
3. 담당자가 Issue의 Human 질문에 직접 답변
4. 구현 AI가 답변 검증·AI 보완 설명·최종 Issue 재작성
5. Human이 최종 Issue를 확인하고 READY 승인
6. 구현 AI가 구현·테스트·Commit·Push·Draft PR 생성
7. 담당자가 PR의 실제 Diff와 테스트를 확인하고 이해도 답변 작성
8. 다른 팀원의 리뷰 AI가 1차 리뷰와 Human 답변 보완 설명 작성
9. 구현 AI가 리뷰를 확인하고 범위 안의 지적을 수정·재검증·Push
10. Human이 최종 코드 확인·Approve·Merge
```

구현 AI의 자체 Diff 검토는 1차 PR 리뷰가 아니다. 1차 리뷰는 PR 작성자가 아닌 다른 팀원의 AI 에이전트가 수행한다.

## 3. Issue 단계

### 3.1 Issue 분석

구현 AI는 Issue, 확정 문서, 관련 코드와 테스트를 확인한다.

확인 대상:

- 목적과 범위·제외 범위
- 확정 정책과 미결정 사항
- 정상·실패·경계 흐름
- API·DB·상태·권한·트랜잭션
- 선행 작업과 도메인 의존성
- 완료 조건과 검증 계획
- 문서·Issue·코드 충돌

Human 결정이 필요하면 Issue의 `Human 착수 검토`에 구체적인 질문을 작성하고 상태를 `HUMAN_ANSWER_REQUIRED`로 남긴 뒤 중단한다.

### 3.2 Human 답변

담당자는 Issue에 직접 답한다. AI가 Human 답변을 대신 작성하지 않는다.

### 3.3 AI 검증과 최종 Issue 재작성

같은 명령을 다시 받으면 구현 AI는 Human 답변을 코드·문서와 대조한다.

- 답변이 충분하면 `AI 보완 설명`을 별도로 작성한다.
- 답변을 반영해 범위·정책·흐름·완료 조건·검증 계획을 최종 계약으로 정리한다.
- 충돌이나 미결정 사항이 남으면 `HOLD`로 남기고 중단한다.
- 구현 가능한 계약이면 `READY_CANDIDATE`로 남긴다.

Human은 최종 계약을 확인한 뒤 READY 체크리스트와 승인자를 작성하고 상태를 `READY`로 변경한다.

## 4. 구현과 Draft PR

`READY` 확인 후 구현 AI는 추가 승인 질문 없이 다음을 수행한다.

```text
기능 브랜치 확인 또는 생성
→ 최소 변경 계획
→ 구현
→ 테스트·직접 검증
→ 실제 Diff 자체 검토
→ 구현 후 이해도 질문 작성
→ Commit·Push
→ develop 대상 Draft PR 생성
```

필수 원칙:

- 승인된 Issue 범위만 구현한다.
- 실행하지 않은 테스트를 PASS로 기록하지 않는다.
- build 실패를 성공으로 표현하지 않는다.
- 정책·API·DB 재결정이 필요하면 중단한다.
- PR에는 실제 변경·실행 결과·미검증 위험만 기록한다.

## 5. PR Human 확인

Draft PR 생성 후 담당자가 다음을 직접 확인한다.

- 실제 Diff와 코드 흐름
- 테스트 명령·결과·실패 로그
- Issue 완료 조건 충족 여부
- 권한·트랜잭션·상태·동시성 위험
- 미검증 범위

담당자는 PR의 구현 후 이해도 질문에 직접 답한다.

## 6. AI 1차 리뷰

PR 작성자가 아닌 다른 팀원의 AI 에이전트가 `PR #번호 1차 리뷰하라` 명령으로 리뷰한다.

필수 입력:

- READY Issue
- 최신 Head SHA와 실제 Diff
- PR 본문
- 코드와 테스트
- 실행·CI 증거
- Human 이해도 답변

리뷰 AI는 결함과 위험을 리뷰하고, Human 답변이 부족하면 답변을 바꾸지 않고 `AI 보완 설명`으로 추가 설명한다.

리뷰 AI는 코드·PR을 수정하거나 Approve·Merge하지 않는다.

## 7. 리뷰 수정

구현 AI가 다시 `Issue #번호 구현하라` 명령을 받으면 해당 Issue의 PR과 1차 리뷰를 확인한다.

자동 수정 가능한 항목:

- Issue 계약 안의 명확한 기능 오류
- 예외 처리·검증·테스트 누락
- 문서·PR 설명 불일치
- 정책 재결정이 필요 없는 BLOCKER·MAJOR·MINOR

자동 수정하지 않는 항목:

- QUESTION
- 범위 확장
- 정책·API·DB·상태·권한·트랜잭션 재결정
- 다른 담당자 계약 변경
- 새로운 라이브러리·인프라 도입

수정 후 테스트·build·직접 검증을 다시 수행하고 Commit·Push와 PR 본문을 갱신한다.

## 8. 최종 Human 확인과 Merge

다음 조건을 충족한 뒤 Human이 최종 확인한다.

- Issue가 READY 상태임
- 완료 조건과 구현·검증 증거가 연결됨
- 담당자가 실제 Diff를 확인하고 답변을 작성함
- 다른 팀원의 AI 1차 리뷰가 완료됨
- 구현 AI가 범위 안의 지적을 수정·재검증함
- 미검증 위험과 미해결 질문이 기록됨
- 해결되지 않은 BLOCKER가 없음
- `docs/GITHUB_RULES.md`의 Human Approve 조건을 충족함

PR 작성자가 Merge한다. AI는 Approve와 Merge를 하지 않는다.

## 9. 실행 재개 원칙

AI 에이전트는 Human이 Issue·PR을 수정한 뒤 스스로 다시 실행되지 않는다.

각 Human 체크포인트 뒤에는 긴 프롬프트 대신 같은 짧은 명령을 다시 사용한다.

```text
Issue #번호 구현하라
```

구현 AI는 처음부터 반복하지 않고 현재 Issue·PR 상태를 읽어 다음 단계부터 재개한다.
