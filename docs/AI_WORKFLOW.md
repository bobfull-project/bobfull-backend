# BobFull AI 협업 워크플로우

## 1. 목적

밥풀 프로젝트에서 AI는 개발자를 대신해 모든 판단을 수행하는 도구가 아니다.

Spring이 반복적인 기반 기능을 추상화해 개발자가 비즈니스 로직에 집중하도록 돕듯이, AI는 반복적인 코드 작성, 코드 탐색, 테스트 초안과 문서화 비용을 줄여 Human 개발자가 다음에 집중하도록 돕는다.

- 해결하려는 문제
- 사용자와 데이터의 전체 흐름
- 도메인 정책과 상태 전이
- API와 DB 구조
- 트랜잭션과 데이터 정합성
- 권한, 동시성, 성능과 장애 위험
- 테스트와 측정 방법
- 구현 결과의 최종 검증

> AI는 코드 작성 비용을 낮춘다.  
> Human은 설계·판단·검증과 결과를 책임진다.

AI를 많이 사용하는 것 자체를 프로젝트 성과로 보지 않는다. AI를 사용해 더 중요한 설계와 문제 해결에 집중했는지를 기준으로 판단한다.

이 문서는 다음 작업에 적용한다.

- Draft Pull Request 작성
- 구현 코드 이해도 확인

## 3. 전체 흐름

```text
AI Issue 초안 작성
→ 담당자 요구사항·설계 검토
→ 관련 도메인 담당자 검토
→ AI 누락·충돌 재검증
→ Human READY 승인

→ 구현 AI 코드·테스트·검증
→ 구현 AI 실제 Diff 자체 검토
→ 구현 AI 이해도 질문 작성
→ Draft PR 작성

→ 담당자 실제 코드 확인과 답변 작성
→ ChatGPT PR 전체 검증
→ Human 리뷰 반영 범위 결정
→ 구현 AI 승인된 항목만 수정
→ ChatGPT 최신 Head 재검토

→ Human 코드 리뷰
→ GitHub Rules의 승인 조건 충족
→ Human Merge
```

각 단계의 목적은 AI 작업을 늘리는 것이 아니라, 요구사항 누락과 이해하지 못한 코드가 Merge되는 것을 줄이는 것이다.

## 4. 역할과 책임

### 4.1 Human 담당자

담당자는 해당 Issue와 구현 결과의 최종 책임자다.

책임:

- 작업 목적과 사용자 흐름 확인
- 서비스 정책과 포함·제외 범위 결정
- API, DB, 상태와 트랜잭션 경계 검토
- AI가 제시한 대안과 위험 검토
- Issue의 `READY` 승인
- 구현 후 실제 코드와 테스트 확인
- 구현 AI가 작성한 이해도 질문에 직접 답변
- ChatGPT 리뷰의 반영 여부 결정
- 남아 있는 위험과 미검증 범위 확인

하지 않는 일:

- AI 답변을 그대로 복사해 자신의 이해도 답변으로 사용
- 실행하지 않은 테스트를 성공으로 승인
- 이해하지 못한 코드를 일정 때문에 그대로 Merge

### 4.2 관련 도메인 담당자

하나의 기능이 여러 담당 영역과 연결되면 관련 담당자가 함께 검토한다.

```text
예약 생성
= 테이블·시간 검증
+ 예약금 결제
+ 예약 생성
+ 최초 참여자 등록
```

```text
예약 참여
= 추가 참여 가능 여부
+ 예약금 결제
+ 참여자 등록
+ 참여 인원 증가
+ 예약 상태 재계산
```

```text
예약 취소
= 참여자 상태 변경
+ 예약 상태 재계산
+ 환불
+ 필요 시 테이블·시간 복구
```

관련 담당자는 자신의 영역과 연결되는 계약만 확인한다. 모든 Issue를 팀 전원이 처음부터 다시 검토하지 않는다.

### 4.3 Issue 설계 AI

ChatGPT 또는 팀이 선택한 설계 AI가 담당할 수 있다.

책임:

- 현재 문서와 코드 분석
- Issue 상세 초안 작성
- 정상·실패·경계 흐름 후보 제시
- 대안과 트레이드오프 제시
- 테스트와 직접 검증 후보 제시
- Human 검토 내용의 누락과 충돌 재검증

하지 않는 일:

- 미확정 정책을 확정된 정책처럼 작성
- Human 결정을 자동으로 대신함
- AI 최초 제안을 최종 계약으로 취급

### 4.4 구현 AI

Codex, Claude Code 또는 팀이 승인한 코드 구현 에이전트를 뜻한다.

책임:

- `READY` Issue의 최종 계약 확인
- 관련 문서, 코드와 테스트 분석
- 승인된 범위의 코드 구현
- 테스트 작성과 실행
- 실제 Diff 자체 검토
- 테스트 결과와 미검증 범위 기록
- 구현 후 Human 이해도 질문 작성
- Commit·Push·Draft PR 작성
- Human이 반영하기로 결정한 리뷰만 수정
- 수정 후 테스트와 검증 재실행

하지 않는 일:

- `DRAFT` Issue 구현
- 미확정 정책 임의 결정
- API나 DB 구조 임의 변경
- Issue 범위 밖 기능 추가
- Human 이해도 답변 대신 작성
- Human 답변 채점 또는 모범 답안 작성
- ChatGPT 리뷰 자동 반영
- Merge

상세 실행과 질문 생성 기준은 `docs/AI_IMPLEMENTATION_GUIDE.md`를 따른다.

### 4.5 ChatGPT 리뷰 AI

책임:

- Issue 계약과 실제 Diff 비교
- 테스트가 완료 조건을 실제로 검증하는지 확인
- 코드, 테스트, 문서와 PR 설명의 불일치 확인
- 권한, 트랜잭션, 정합성, 동시성과 실패 위험 검토
- Human 이해도 답변과 실제 코드 비교
- 리뷰 수정 후 최신 Head 재검토

하지 않는 일:

- Human의 리뷰 반영 범위 결정
- 코드 자동 수정
- 구현 AI에게 승인되지 않은 수정 확정
- Merge

### 4.6 Human 리뷰어

책임:

- 코드와 팀 규칙 검토
- ChatGPT 리뷰 참고
- 리뷰 항목에 대한 의견 제시
- GitHub Rules에 따른 Approve
- Merge 가능 여부 판단

AI 리뷰는 Human 리뷰를 대체하지 않는다.

## 5. Issue 초안과 READY

### 5.1 Issue 초안

Issue에는 가능한 범위에서 다음 내용을 작성한다.

- 문제와 목적
- 현재 확인된 사실
- 구현 범위
- 제외 범위
- 정상 흐름
- 실패 흐름
- 중요한 경계 조건
- 완료 조건
- 테스트와 직접 검증 방법
- 관련 도메인과 담당자
- 미결정 사항
- 대안과 선택 이유

AI가 상세 초안을 작성할 수 있지만, AI 초안은 확정 계약이 아니다.

### 5.2 Human 검토

담당자는 구현 전에 다음을 자신의 말로 확인한다.

- 무엇을 해결하려는가
- 요청과 데이터는 어떻게 흐르는가
- 어떤 상태가 변경되는가
- 실패하면 어떤 결과가 되어야 하는가
- 트랜잭션과 권한 경계는 어디인가
- 테스트로 무엇을 확인할 것인가
- 이번 Issue에 포함하지 않는 것은 무엇인가

구현 전 질문은 어려운 기술 시험이 아니다.

질문은 다음 수준을 기준으로 한다.

- 신입 백엔드 기술면접 수준
- 해당 구현에서 반드시 알아야 하는 핵심
- 실제 설계와 연결된 내용
- 담당자가 구현 방향을 이해했는지 확인할 수 있는 내용

다음은 요구하지 않는다.

- 사용하지 않는 기술의 깊은 이론
- Spring, JVM, DB 내부 구현의 지나친 세부 사항
- 명령어와 설정값 암기
- 실제 구현과 관계없는 비교 질문
- 담당자를 곤란하게 만들기 위한 함정 질문

### 5.3 READY 조건

다음이 완료되면 담당자가 Issue를 `READY`로 승인한다.

- 서비스 정책이 결정됨
- 구현 범위와 제외 범위가 확정됨
- 완료 조건이 확정됨
- 테스트와 직접 검증 방향이 정해짐
- API 또는 DB 결정이 필요한 경우 합의됨
- 관련 담당자 검토가 필요한 경우 완료됨
- 핵심 미결정 사항이 남아 있지 않음
- 담당자가 구현 시작을 승인함

`READY`는 구현 시작 승인이다.

구현 AI는 영향 범위와 작업 계획을 보고할 수 있지만, 승인된 범위를 구현하기 위해 같은 허가를 반복해서 요청하지 않는다.

## 6. 구현과 Draft PR

구현 AI의 기본 실행 흐름은 다음과 같다.

```text
READY Issue 확인
→ 관련 규칙·코드·테스트 확인
→ 영향 범위 확인
→ 최소 변경 계획
→ 구현
→ 테스트
→ 실제 Diff 자체 검토
→ 검증 결과와 미검증 범위 정리
→ 구현 후 이해도 질문 작성
→ Commit·Push
→ Draft PR
```

필수 원칙:

- 한 번에 하나의 `READY` Issue만 처리한다.
- Issue 범위 안에서 최소한으로 변경한다.
- 불필요한 리팩터링을 함께 수행하지 않는다.
- Git 작업은 `docs/GITHUB_RULES.md`를 따른다.
- 구현은 `docs/CODE_CONVENTION.md`를 따른다.
- 실행하지 않은 테스트를 성공으로 기록하지 않는다.
- 테스트 통과와 요구사항 전체 충족을 구분한다.
- 일반 구현 오류는 승인된 범위 안에서 수정한다.
- 설계나 정책 재결정이 필요하면 중단한다.

## 7. 구현 후 Human 이해도 질문

구현 AI는 구현과 테스트가 끝난 뒤 최신 실제 Diff를 기준으로 질문을 작성한다.

질문 목적은 담당자를 시험하는 것이 아니다. 담당자가 자신이 구현한 기능의 목적, 코드 흐름, 실패 조건과 검증 범위를 설명할 수 있는지 확인하는 것이 목적이다.

### 질문 기준

- 대상 Issue와 실제 Diff에 근거한다.
- 구현한 내용 중 반드시 알아야 하는 핵심만 묻는다.
- 신입 백엔드 기술면접 수준을 기준으로 한다.
- 어려운 이론보다 실제 동작과 선택 이유를 우선한다.
- 질문 수를 채우기 위한 불필요한 질문을 만들지 않는다.

### 우선 질문 관점

- 이 기능이 해결한 문제
- 요청부터 응답까지의 흐름
- 주요 클래스와 메서드의 역할
- 데이터나 Entity 상태 변경 과정
- 트랜잭션 범위와 실패 결과
- 인증·인가 검증 위치
- 동시 요청에서 발생 가능한 문제
- 테스트가 보장하는 내용
- 테스트가 보장하지 못하는 내용
- 구현 방식의 가장 중요한 선택 이유

### 질문 난이도 제한

다음 질문은 만들지 않는다.

- 이번 구현에서 사용하지 않은 기술
- JVM, Spring, Hibernate, DB의 지나치게 깊은 내부 구현
- 라이브러리 내부 알고리즘 설명
- 클래스명, 명령어, 설정값 암기
- 실제 Diff와 관계없는 기술 비교
- 한 내용을 지나치게 잘게 나눈 반복 질문
- 함정형 질문

중요한 기능이라고 해서 이론 난이도를 높이지 않는다. 대신 실패 조건, 데이터 정합성, 권한, Rollback, 동시성과 테스트의 미검증 범위를 더 확인한다.

### 질문 수

- 문서·설정·작은 변경: 1~2개
- 일반 CRUD·기능 구현: 3~5개
- 결제·환불·좌석·권한·동시성·상태 전이: 4~6개

필요한 핵심을 확인할 수 있다면 최소 개수로 작성한다.

구현 AI는 질문만 작성하며 다음을 하지 않는다.

- Human 답변 대신 작성
- 모범 답안 작성
- 답변 채점
- PASS 또는 FAIL 판정
- 답변 자동 보완
- 이해도 답변을 이유로 승인되지 않은 코드 수정

상세 기준은 `docs/AI_IMPLEMENTATION_GUIDE.md`를 따른다.

## 8. Human 이해도 답변

담당자는 실제 코드와 테스트를 읽은 뒤 직접 답변한다.

답변에는 가능한 범위에서 다음을 포함한다.

- 구현 목적
- 핵심 코드와 데이터 흐름
- 상태 변경 과정
- 가장 중요한 실패 조건
- 트랜잭션·권한·동시성 경계
- 테스트가 보장하는 범위
- 테스트가 보장하지 않는 범위
- 남아 있는 위험

모르는 내용은 모른다고 기록할 수 있다. 모른다는 사실 자체보다 모르면서 이해한 것처럼 작성하는 것을 더 큰 문제로 본다.

Human 답변은 구현 AI가 수정하거나 덮어쓰지 않는다.

## 9. 기본 검토와 강화 검토

모든 작업에 같은 무게의 절차를 적용하지 않는다.

### 9.1 기본 검토

대상 예시:

- 문서 변경
- 작은 설정 변경
- 단순 조회
- 낮은 영향의 CRUD
- 작은 리팩터링

기본 절차:

```text
구현 AI 자체 Diff 검토
→ 담당자 코드 확인과 이해도
→ ChatGPT PR 검토
→ CI
→ Human 코드 리뷰와 Approve
```

### 9.2 강화 검토

대상 예시:

- 결제·환불·지급 예정 금액
- 예약·좌석·참여 인원
- 트랜잭션과 동시성
- 인증·인가
- 예약 상태 전이
- DB 구조 변경
- 채팅 접근 권한
- 데이터 손실이나 운영 장애 가능성이 큰 설정

추가 확인:

- 구현 전 대안과 실패 시나리오
- 정상·실패·경계·Rollback 테스트
- 필요한 직접 검증
- 요구사항·코드·테스트 연결 확인
- 관련 도메인 담당자 검토
- 수정 후 최신 Head 재검토

강화 검토는 질문 난이도를 높이는 절차가 아니다. 중요한 실패와 검증 범위를 더 확인하는 절차다.

---

## 10. 검증과 증거

PR에는 다음을 구분해 기록한다.

- 실행한 명령
- 확인된 실제 결과
- 테스트가 보장하는 범위
- 테스트가 보장하지 않는 범위
- 직접 확인한 내용
- 실행하지 못한 검증
- 미검증 항목
- 남아 있는 위험

다음과 같이 과장하지 않는다.

```text
테스트 성공
≠ 요구사항 전체 충족

단위 테스트 성공
≠ 실제 DB·외부 환경 검증

기능 동작
≠ 동시성 안전

응답 시간 1회 확인
≠ 성능 개선 증명
```

검증하지 못한 항목은 실패가 아니라 `미검증`으로 기록한다.

---

## 11. ChatGPT PR 리뷰

ChatGPT는 다음 순서로 검토한다.

```text
Issue 계약
→ 실제 변경 Diff
→ 테스트 코드와 실행 결과
→ 문서와 PR 설명
→ Human 이해도 답변
→ CI와 최신 Head
```

주요 검토 항목:

- Issue 범위가 실제 코드에 반영됐는가
- 범위 밖 변경이 포함됐는가
- 정상·실패·경계 조건이 빠지지 않았는가
- 권한과 소유권 검증이 올바른가
- 트랜잭션과 Rollback 결과가 요구사항과 맞는가
- 좌석·금액·상태 정합성 위험이 없는가
- 테스트가 완료 조건을 실제로 검증하는가
- 실행하지 않은 검증을 성공으로 기록하지 않았는가
- Human 답변과 실제 코드가 일치하는가
- 중대한 미이해나 잘못된 설명이 남아 있는가

ChatGPT는 발견 사항을 근거와 함께 리뷰로 남긴다.

---

## 12. 리뷰 반영 결정

ChatGPT 리뷰가 작성되면 Human 담당자 또는 리뷰어가 항목별로 다음 중 하나를 결정한다.

| 결정 | 의미 |
|---|---|
| `반영` | 현재 PR에서 수정 |
| `미반영` | 수정하지 않으며 이유 기록 |
| `후속 Issue` | 현재 범위를 넘으므로 별도 처리 |

미반영과 후속 Issue에는 가능한 범위에서 다음 근거를 작성한다.

- 현재 Issue 범위
- 일정과 우선순위
- 위험도
- 대안
- 현재 남는 위험

구현 AI는 `반영`으로 승인된 항목만 수정한다.

---

## 13. 수정과 최신 Head 재검토

```text
Human 리뷰 결정
→ 구현 AI 승인 항목 수정
→ 테스트와 검증 재실행
→ Push
→ ChatGPT 최신 Head 재검토
→ Human 최종 리뷰
```

수정 후에는 이전 리뷰 시점의 코드가 아니라 최신 Commit을 기준으로 판단한다.

수정 과정에서 새로운 설계 결정이 필요해지면 구현 AI는 임의로 범위를 확장하지 않고 중단한다.

---

## 14. 중단 조건

구현 AI는 다음 상황에서 작업을 중단하고 Human 판단을 요청한다.

- Issue와 문서 또는 현재 코드가 충돌함
- 핵심 서비스 정책이 확정되지 않음
- API 또는 DB 구조 재결정이 필요함
- 다른 담당자 영역의 계약 변경이 필요함
- 결제, 좌석, 권한 또는 데이터 손실 위험이 발견됨
- 테스트를 삭제하거나 약화해야 함
- Issue 범위 밖 변경이 필요함
- 핵심 검증에 필요한 환경이나 권한이 없음
- 다른 작업자의 변경을 덮어쓸 위험이 있음

보고할 때 다음을 구분한다.

- 발생한 문제
- 확인된 사실
- 추정 원인
- 영향 범위
- 가능한 선택지
- 권장 방향
- 필요한 Human 결정

확인되지 않은 추정을 사실처럼 보고하지 않는다.

---

## 15. Merge 조건

다음 조건을 확인한 뒤 Human이 Merge한다.

- Issue 완료 조건 충족
- 실행한 테스트와 결과 기록
- 미실행·미검증 범위 기록
- 담당자가 실제 코드를 확인하고 이해도 답변 작성
- ChatGPT PR 검토 완료
- Human이 리뷰 반영 범위 결정
- 승인된 수정 반영
- 최신 Head와 CI 재확인
- GitHub Rules의 Approve 조건 충족
- 범위 밖 변경 없음
- 남은 위험을 Human이 확인함

AI는 Merge하지 않는다.

Merge 주체와 승인 인원은 `docs/GITHUB_RULES.md`를 따른다.

---

## 16. 적용하지 않는 자동화

초기 워크플로우에서는 다음을 적용하지 않는다.

- Issue부터 Merge까지 완전 자동화
- 여러 구현 AI가 같은 Issue를 동시에 수정
- 모든 PR에 여러 AI 리뷰 강제
- Human 이해도 답변 자동 작성
- Human 답변 자동 채점
- AI 리뷰 자동 반영
- 무제한 자동 수정 반복
- AI 자동 Merge
- 실제 필요성이 없는 복잡한 Hook과 Agent 구성

자동화는 실제 반복 문제와 효과가 확인된 경우에만 추가한다.

---

## 17. 시험 운영과 변경

이 워크플로우는 처음부터 완성된 규칙으로 취급하지 않는다.

첫 주에는 작은 기능에 시험 적용하고 다음을 확인한다.

- Issue 작성 시간이 지나치게 길지 않은가
- 구현 AI의 요구사항 누락이 줄었는가
- 담당자가 실제 코드를 설명할 수 있는가
- 질문 수준이 지나치게 어렵지 않은가
- PR 리뷰 시간이 줄었는가
- 불필요한 문서와 반복 절차가 생기지 않는가
- 구현과 학습 시간을 방해하지 않는가

문제가 확인되면 절차를 무조건 추가하지 않는다.

```text
불필요한 항목 제거
→ 중복 문서 통합
→ 질문 수와 검토 범위 축소
→ 필요한 경우에만 절차 추가
```

팀이 합의한 변경은 문서에 반영한다.

---

## 18. 최종 운영 원칙

```text
AI는 상세한 초안, 구현과 검증 근거를 제공한다.

Human은 정책, 설계, 트레이드오프와
리뷰 반영 범위를 결정한다.

READY는 구현 시작 승인이다.

구현 AI는 승인된 계약만 구현하고
코드·테스트·질문·Draft PR까지 인계한다.

이해도 질문은 실제 Diff를 기준으로 하며
신입 백엔드 기술면접 수준을 넘지 않는다.

ChatGPT는 PR 전체와 최신 Head를 독립적으로 검토한다.

테스트 통과와 요구사항 충족,
운영 환경 보장을 구분한다.

AI가 작성한 코드도 담당자가 설명할 수 있어야 한다.

Merge와 결과의 최종 책임은 Human에게 있다.
```
