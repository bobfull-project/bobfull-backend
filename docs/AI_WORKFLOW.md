# BobFull AI 협업 워크플로우

## 1. 목적

AI는 반복적인 코드 작성·탐색·테스트 초안·문서화 비용을 줄이고, Human은 설계·판단·검증과 결과를 책임진다.

> 코드 타이핑은 AI와 함께하고, 설계와 판단은 사람이 책임진다.

AI 사용량 자체를 성과로 보지 않는다. 요구사항 누락을 줄이고 더 중요한 설계·정합성·성능·장애 대응에 집중했는지를 본다.

## 2. 문서 책임

| 목적 | 기준 문서 |
|---|---|
| 프로젝트 확정 정책 | `docs/PROJECT_CONTEXT.md` |
| AI 공통 진입점·금지사항 | `AGENTS.md` |
| 구현 AI 실행 | `docs/AI_IMPLEMENTATION_GUIDE.md` |
| ChatGPT PR 리뷰 | `docs/AI_REVIEW_GUIDE.md` |
| 테스트 작성·증거 | `docs/TEST_CONVENTION.md` |
| Git·PR·Merge | `docs/GITHUB_RULES.md` |
| Issue·PR 기록 형식 | Issue·PR 템플릿 |

문서가 충돌하면 확정 정책은 `docs/PROJECT_CONTEXT.md`, 작업 방식은 각 책임 문서를 우선한다.

## 3. 전체 흐름

```text
AI가 Issue 초안 작성
→ Human 요구사항·정책·완료 조건 검토
→ 관련 도메인 영향 확인
→ Human READY 승인

→ 구현 AI가 코드·테스트·검증 수행
→ 실제 Diff 자체 검토
→ 이해도 질문 작성
→ Draft PR 작성

→ 담당자가 실제 코드 확인과 답변 작성
→ ChatGPT가 Issue·Diff·테스트·답변 리뷰
→ Human이 리뷰 반영 범위 결정
→ 구현 AI가 승인된 내용만 수정·재검증
→ ChatGPT가 최신 Head 재검토

→ Human 코드 리뷰·Approve
→ PR 작성자가 Merge
```

## 4. 역할과 책임

### Human 담당자

- 문제·정책·포함·제외 범위를 결정한다.
- API·DB·상태·트랜잭션·권한 경계를 검토한다.
- Issue를 `READY`로 승인한다.
- 구현 후 실제 Diff와 테스트 결과를 확인한다.
- 이해도 질문에 직접 답한다.
- 리뷰 반영 범위와 Merge를 결정한다.

실행하지 않은 테스트를 승인하거나 이해하지 못한 코드를 Merge하지 않는다.

### 관련 도메인 담당자

여러 영역이 연결되는 Issue는 `docs/DOMAIN_DEPENDENCIES.md`를 기준으로 자신의 계약과 실패 결과를 확인한다. 모든 Issue를 팀 전원이 처음부터 검토하지 않는다.

### Issue 설계 AI

- 현재 정책·문서·코드를 분석한다.
- 구현 범위, 실패·경계 흐름, 완료 조건과 검증 후보를 작성한다.
- 누락·충돌·대안을 제시한다.

미확정 정책을 확정하거나 Human 결정을 대신하지 않는다.

### 구현 AI

- `READY` Issue만 구현한다.
- 승인 범위 안에서 코드·테스트·검증·문서·Draft PR을 작성한다.
- 실제 실행 결과와 미검증 범위를 구분한다.
- 최신 Diff 기반 이해도 질문을 남긴다.
- 승인된 리뷰 항목만 수정하고 다시 검증한다.

정책·API·DB를 임의로 바꾸거나 Human 답변·Merge를 대신하지 않는다.

### ChatGPT 리뷰 AI

- Issue 계약과 최신 Diff를 대조한다.
- 테스트가 완료 조건과 위험을 실제로 검증하는지 확인한다.
- 정합성·권한·트랜잭션·동시성·실패 위험을 검토한다.
- Human 답변과 실제 코드를 비교한다.
- 수정 후 최신 Head와 CI를 다시 확인한다.

코드 자동 수정, 리뷰 반영 결정, 승인과 Merge는 하지 않는다.

### Human 리뷰어

AI 리뷰를 참고해 실제 코드를 검토하고 `docs/GITHUB_RULES.md`에 따라 Approve한다. AI 리뷰는 Human 리뷰를 대체하지 않는다.

## 5. Issue와 READY

Issue는 필요한 범위에서 다음을 포함한다.

- 문제와 목적
- 확인된 사실과 확정 정책
- 구현 범위와 제외 범위
- 정상·실패·경계 흐름
- 완료 조건
- 테스트·직접 검증 방법
- 관련 도메인·선행·후속 작업
- 미결정 사항
- 중요한 선택과 근거

단순 작업은 불필요한 항목을 생략한다.

다음 조건이 충족되면 Human이 `READY`로 승인한다.

- 정책과 범위가 확정됨
- 완료 조건과 검증 방향이 정해짐
- 필요한 API·DB 합의가 완료됨
- 관련 담당자 검토가 완료됨
- 구현을 막는 미결정 사항이 없음

`READY`는 구현 시작 승인이다. 구현 AI는 같은 허가를 반복 요청하지 않는다.

## 6. 구현과 Draft PR

구현 AI는 다음 순서로 작업한다.

```text
READY 계약 확인
→ 관련 정책·코드·테스트 확인
→ 영향 범위와 최소 변경 계획
→ 구현
→ 테스트·직접 검증
→ 실제 Diff 자체 검토
→ 이해도 질문과 검증 증거 정리
→ Commit·Push·Draft PR
```

필수 원칙:

- 한 번에 하나의 승인된 Issue만 처리한다.
- 범위 밖 기능과 불필요한 리팩터링을 추가하지 않는다.
- 요구사항에 없는 기술을 임의로 도입하지 않는다.
- 테스트 통과와 요구사항 충족을 구분한다.
- 정책·설계 재결정이 필요하면 중단한다.

상세 실행 기준은 `docs/AI_IMPLEMENTATION_GUIDE.md`를 따른다.

## 7. 테스트와 검증 증거

테스트 코드는 `docs/TEST_CONVENTION.md`를 따른다.

PR에는 실제 수행한 내용만 기록한다.

- 작성한 테스트와 작성 이유
- 한글 설명형 테스트명과 `given → when → then`
- 실행 명령과 환경
- `PASS | FAIL | HOLD | NOT_RUN`
- 통과 건수 또는 실패 로그
- 직접 검증 결과
- 보장 범위와 보장하지 못한 범위
- 최신 검증 대상 Commit과 CI 상태

실행하지 않은 테스트를 `PASS`로 기록하지 않는다. 테스트가 통과해도 트랜잭션·락·Rollback을 실제 조건으로 증명하지 못했다면 미검증으로 남긴다.

## 8. 구현 후 Human 이해도

구현 AI는 최신 Issue·Diff·테스트에 근거한 질문만 작성한다.

질문 범위:

- 기능 목적과 요청·데이터 흐름
- 주요 클래스의 역할
- 상태 변경과 트랜잭션 경계
- 실패 시 결과
- 권한·소유권·동시성 위험
- 테스트가 보장하는 것과 보장하지 못하는 것
- 중요한 선택의 이유

질문 수 기준:

- 작은 변경: 1~2개
- 일반 기능: 3~5개
- 결제·좌석·권한·동시성 등 중요 기능: 4~6개

신입 백엔드 기술면접 수준을 유지한다. 구현 AI는 답변·채점·모범 답안을 작성하지 않는다.

## 9. 리뷰와 수정

모든 기능 PR은 ChatGPT 리뷰를 거친다.

### 기본 검토

- Issue 범위와 완료 조건
- 기능 오류와 예외 처리
- 테스트·실행 증거
- 문서·PR 설명 정합성
- 비밀정보와 불필요한 복잡성

### 강화 검토

결제·환불·정산·예약·좌석·상태 변경·권한·동시성·외부 연동은 다음을 추가한다.

- 트랜잭션·Rollback·부분 성공
- 중복 요청·멱등성
- 경쟁 조건·락·재시도
- 금액·수량·상태 정합성
- 권한·소유권
- 외부 연동 실패 보상

중요 기능은 검증 범위를 넓히되 질문의 이론 난이도를 높이지 않는다.

Human이 반영할 항목을 결정하면 구현 AI가 해당 내용만 수정하고 테스트·직접 검증을 다시 수행한다. ChatGPT는 최신 Head를 기준으로 기존 지적 해결 여부와 새 위험을 재검토한다.

## 10. Merge 조건

다음이 충족된 뒤 PR 작성자가 Merge한다.

- Issue 완료 조건과 구현·검증 증거가 연결됨
- 담당자가 실제 Diff를 확인하고 이해도 답변을 작성함
- ChatGPT 최신 Head 리뷰가 완료됨
- 미검증 범위와 남은 위험이 기록됨
- `docs/GITHUB_RULES.md`의 Human Approve 조건을 충족함

AI는 Merge하지 않는다.

## 11. 즉시 중단 조건

다음 상황에서는 구현·수정을 멈추고 Human 판단을 요청한다.

- 확정 정책이나 문서가 충돌함
- 미확정 정책 결정이 필요함
- API·DB·트랜잭션 경계를 다시 결정해야 함
- 다른 담당자 영역의 계약 변경이 필요함
- 데이터 손실·정합성·권한·보안 위험이 있음
- 테스트 삭제·약화 또는 범위 밖 변경이 필요함
- 핵심 검증 환경이나 권한이 없음

API Key, 비밀번호, JWT, 운영 환경변수, 실제 개인정보와 식별 가능한 운영 데이터는 AI 입력에 포함하지 않는다.
