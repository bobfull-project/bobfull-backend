# BobFull AI PR 리뷰 가이드

## 1. 목적

이 문서는 ChatGPT가 밥풀 프로젝트의 기능 PR을 검토할 때 따르는 기준이다.

모든 기능 PR은 Human 코드 리뷰 전에 AI 전체 검토를 거친다.
AI 리뷰의 목적은 코드를 대신 승인하는 것이 아니라, Issue 계약과 실제 구현·테스트·설명이 서로 일치하는지 확인하는 것이다.

> AI는 누락·충돌·위험을 찾는다.  
> Human은 코드 이해와 최종 승인·Merge를 책임진다.

---

## 2. 리뷰 입력

AI는 가능한 범위에서 다음 자료를 함께 확인한다.

1. READY 상태의 대상 Issue
2. PR 본문
3. 최신 Head의 실제 Diff
4. 변경 파일 전체
5. 코드 리뷰 가이드와 읽는 순서
6. 작성한 테스트 코드와 작성 이유
7. 실제 실행 명령·결과·증거
8. 미검증 범위와 남은 위험
9. Human 이해도 답변
10. 최신 Commit SHA와 GitHub Actions 결과

자료가 없으면 추정하지 않고 `미확인`으로 기록한다.

---

## 3. 리뷰 순서

```text
Issue 계약과 완료 조건
→ PR 범위와 실제 Diff
→ 요청부터 응답까지 코드 흐름
→ 상태·금액·좌석·권한·트랜잭션
→ 테스트 코드와 실행 증거
→ Human 이해도 답변
→ 문서·API·DB 계약
→ 최신 Commit과 CI
```

PR 본문의 코드 읽기 순서를 참고하되, 실제 Diff와 다르면 실제 코드를 기준으로 검토한다.

---

## 4. 기본 검토 항목

### 4.1 Issue 범위

- Issue가 READY 상태에서 구현됐는가
- 구현 범위가 완료 조건과 일치하는가
- 제외 범위의 기능이 섞이지 않았는가
- 불필요한 리팩터링이나 다른 담당자 영역 변경이 없는가
- 미확정 정책을 AI가 임의로 결정하지 않았는가

### 4.2 코드 흐름

- Controller가 요청·인증·응답만 처리하는가
- Service가 핵심 유스케이스와 트랜잭션을 담당하는가
- Entity 상태 변경이 의미 있는 메서드로 표현됐는가
- Repository 조회·저장 조건이 요구사항과 일치하는가
- 요청부터 응답까지 데이터가 누락되거나 중복 전달되지 않는가
- PR의 코드 읽기 순서와 실제 호출 흐름이 일치하는가

### 4.3 오류와 예외

- 존재하지 않는 자원, 잘못된 상태, 중복 요청이 올바르게 실패하는가
- 예외 발생 후 데이터가 일부만 저장되지 않는가
- 예외를 숨기거나 성공 응답으로 처리하지 않는가
- ErrorCode와 HTTP 응답이 팀 규칙과 일치하는가

### 4.4 인증과 권한

- 인증 사용자 ID를 요청 Body 값으로 신뢰하지 않는가
- 소유자·사장님·관리자 권한을 서버에서 확인하는가
- 다른 사용자의 예약·결제·채팅에 접근할 수 없는가
- 권한 검증이 누락되거나 Controller에 반복 구현되지 않았는가

### 4.5 트랜잭션과 Rollback

- 함께 성공하거나 실패해야 하는 작업이 같은 트랜잭션에 있는가
- 중간 실패 시 최종 DB 상태가 요구사항과 일치하는가
- 외부 호출과 DB 트랜잭션의 순서가 위험하지 않은가
- 트랜잭션 범위를 지나치게 넓히거나 긴 대기를 포함하지 않는가

### 4.6 데이터 정합성

다음 값은 특히 집중해서 확인한다.

- 예약 상태와 참여자 상태
- 테이블 정원, 현재 참여 인원, 잔여 인원
- 예약금, 환불액, 지급 예정 금액
- 방문·노쇼 상태
- 중복 예약과 중복 참여
- 취소 후 좌석·상태 복구

계산 결과와 DB 상태가 서로 어긋날 가능성을 확인한다.

### 4.7 동시성·중복 요청

해당 기능에 동시 요청 가능성이 있으면 다음을 확인한다.

- 사전 조회만으로 중복을 막고 있지 않은가
- 마지막 좌석에 여러 요청이 들어와도 정원을 초과하지 않는가
- 동일 테이블·시간에 예약이 중복 생성되지 않는가
- DB 제약조건, 락, 원자적 갱신 또는 예외 처리가 실제로 연결되는가
- 동시성 테스트가 실제 경쟁 상황을 만들고 최종 상태를 검증하는가

---

## 5. 테스트 리뷰

### 5.1 테스트 코드 형식

- 테스트 메서드명이 한글 설명형인가
- 이름에 조건과 기대 결과가 드러나는가
- 본문이 `given → when → then` 순서인가
- 하나의 테스트가 하나의 핵심 결과를 검증하는가
- 구현 세부사항보다 완료 조건과 위험을 검증하는가

테스트 규칙은 `docs/TEST_CONVENTION.md`를 따른다.

### 5.2 작성 이유와 계약 연결

각 테스트에 대해 다음을 확인한다.

- 왜 이 테스트가 필요한가
- 어떤 Issue 완료 조건 또는 실패 위험을 검증하는가
- 기대 결과가 실제 assertion으로 확인되는가
- 테스트명과 테스트 내용이 일치하는가
- 의미 없는 Mock 호출 횟수만 확인하고 핵심 상태를 놓치지 않는가

### 5.3 실제 실행 증거

- 실제 실행한 명령이 기록됐는가
- 테스트 클래스·메서드 또는 통과 건수가 증거로 남았는가
- 실행하지 않은 테스트를 PASS로 기록하지 않았는가
- 최신 Commit을 대상으로 실행한 결과인가
- 로컬 결과와 GitHub Actions 결과를 구분했는가
- 테스트가 보장하지 않는 범위를 함께 기록했는가

### 5.4 테스트 결과 해석

다음을 자동으로 동일시하지 않는다.

```text
단위 테스트 성공 ≠ 실제 DB·외부 시스템 검증
전체 테스트 성공 ≠ 동시성 안전
동시성 테스트 1회 성공 ≠ 운영 환경 완전 보장
로컬 응답시간 개선 ≠ 운영 성능·SLA 증명
빌드 성공 ≠ 기능 요구사항 충족
```

## 6. 리뷰어 확인 포인트 검증

PR 작성자가 제시한 리뷰어 확인 포인트를 실제 코드와 대조한다.

- 중요한 위험이 빠지지 않았는가
- 단순 파일 목록이 아니라 실제 실패 가능성을 안내하는가
- 확인할 클래스·메서드·테스트가 정확한가
- 미검증 위험을 숨기지 않았는가

AI는 PR 작성자의 확인 포인트 외에도 중대한 위험을 발견하면 추가로 리뷰한다.

---

## 7. Human 이해도 답변 검토

AI는 Human 답변을 시험 점수처럼 평가하지 않는다.

다음만 확인한다.

- 답변이 실제 코드와 일치하는가
- 핵심 데이터 흐름을 반대로 이해하지 않았는가
- 트랜잭션·권한·동시성 보호 대상을 잘못 이해하지 않았는가
- 테스트가 보장하지 않는 범위를 과대평가하지 않았는가
- 금액·좌석·상태 정합성에 사고로 이어질 오해가 없는가

단순 용어 부족이나 표현 미숙은 차단 사유로 보지 않는다.
중대한 오해가 있으면 코드 근거와 함께 보완 설명을 제공하되, Human 원문을 수정하거나 대신 작성하지 않는다.

---

## 8. 강화 검토 대상

다음 PR은 기본 검토보다 실패·정합성·검증 증거를 더 자세히 확인한다.

- 결제·환불·지급 예정 금액
- 예약·좌석·참여 인원
- 인증·인가
- 트랜잭션·Rollback
- 동시성·중복 요청
- 상태 전이
- DB 구조·제약조건 변경
- 채팅 접근 권한
- 캐시·비동기·알림의 원본 데이터 영향
- 데이터 손실이나 운영 장애 가능성이 큰 설정

강화 검토는 어려운 이론 질문을 추가하는 절차가 아니다.
실패 시 최종 상태와 증거를 더 엄격하게 확인하는 절차다.

## 9. 리뷰 결과 작성 방식

리뷰는 중요한 문제부터 작성한다.

### 분류

| 분류 | 의미 |
|---|---|
| `필수 수정` | 버그, 보안, 데이터 정합성, 권한, 완료 조건 누락처럼 Merge 전에 해결해야 하는 문제 |
| `확인 필요` | 설계 의도·검증 증거·정책이 불명확해 Human 확인이 필요한 항목 |
| `개선 제안` | 현재 기능은 동작하지만 가독성·유지보수성 개선에 도움이 되는 비차단 의견 |

### 리뷰 항목 형식

```text
[분류] 문제 제목

위치:
확인된 사실:
영향:
근거:
권장 조치:
```

- 가능한 경우 파일·클래스·메서드 위치를 적는다.
- 취향 차이를 필수 수정으로 과장하지 않는다.
- 근거 없는 가능성만으로 결함을 단정하지 않는다.
- 같은 원인의 문제는 여러 항목으로 쪼개 반복하지 않는다.

---

## 10. 리뷰 완료 요약

리뷰 마지막에는 다음을 간단히 정리한다.

```text
필수 수정:
확인 필요:
개선 제안:

테스트·검증 상태:
미검증 위험:
Human 이해도 중대한 오해:
최신 검토 Commit:
```

필수 수정 사항이 없으면 `확인된 필수 수정 없음`으로 기록한다.
이는 코드가 완벽하거나 운영 안전성이 모두 증명됐다는 뜻이 아니다.

---

## 11. 수정 후 최신 Head 재검토

구현 AI가 수정사항을 Push하면 반드시 최신 Commit을 다시 검토한다.

재검토 항목:

1. 이전 필수 수정이 실제로 해결됐는가
2. 수정으로 새로운 범위 이탈이나 회귀가 생기지 않았는가
3. 관련 테스트가 다시 실행됐는가
4. PR의 테스트 증거와 미검증 범위가 갱신됐는가
5. 코드 읽기 순서와 리뷰어 확인 포인트가 최신 Diff와 일치하는가
6. GitHub Actions가 최신 Commit을 대상으로 실행됐는가

이전 Commit의 리뷰 결과를 최신 Head에 그대로 적용하지 않는다.

---

## 12. AI 리뷰의 금지 사항

AI는 다음을 하지 않는다.

- PR 자동 승인
- Merge
- Human 이해도 답변 대신 작성
- Human 답변 점수화 또는 재시험 요구
- 취향 차이를 결함으로 단정
- 실행하지 않은 테스트를 성공으로 판단
- 미검증 내용을 검증 완료로 표현
- Issue 범위 밖 대규모 리팩터링 요구
- 최신 Diff 확인 없이 이전 리뷰를 최종 결과로 사용

최종 Approve와 Merge는 Human 리뷰어와 PR 작성자가 `docs/GITHUB_RULES.md`에 따라 수행한다.
