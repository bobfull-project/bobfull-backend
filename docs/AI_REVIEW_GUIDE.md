# BobFull AI PR 리뷰 가이드

## 1. 목적

모든 기능 PR은 Human 코드 리뷰 전에 ChatGPT 검토를 거친다.

AI 리뷰는 코드를 승인하는 단계가 아니다. READY Issue, 최신 Diff, 테스트·검증 증거, PR 설명과 Human 답변이 서로 일치하는지 확인해 누락·충돌·위험을 찾는다.

> AI는 위험을 찾고, Human은 승인과 Merge를 책임진다.

## 2. 필수 입력

가능한 범위에서 다음을 함께 확인한다.

1. READY 상태의 대상 Issue
2. `docs/PROJECT_CONTEXT.md`와 관련 정책 문서
3. PR 본문과 코드 읽는 순서
4. 최신 Head의 실제 Diff
5. 관련 코드와 테스트
6. 테스트·직접 검증·CI 증거
7. 미검증 범위와 남은 위험
8. 담당자의 이해도 답변

자료가 없거나 오래됐으면 추정하지 않고 `확인할 수 없음`으로 기록한다. 항상 최신 Head를 검토한다.

## 3. 검토 수준

### 기본 검토

모든 기능 PR에서 확인한다.

- Issue 범위와 완료 조건
- 기능 오류·입력 검증·예외 처리
- 코드와 PR 설명의 일치
- 테스트가 완료 조건을 실제로 검증하는지
- 실행 결과와 미검증 범위
- 비밀정보·개인정보 포함 여부
- 범위 밖 기능·불필요한 복잡성

### 강화 검토

결제·환불·정산·예약·좌석·상태 변경·권한·동시성·외부 연동에 추가 적용한다.

- 트랜잭션·Rollback·부분 성공
- 중복 요청·멱등성
- 경쟁 조건·락·재시도
- 금액·수량·상태 정합성
- 권한·소유권
- 외부 연동 실패와 보상

중요 기능은 검증 범위를 넓히되 불필요한 이론 질문을 늘리지 않는다.

## 4. 리뷰 순서

### 4.1 계약 확인

- Issue가 `READY`였는가
- 구현 범위와 제외 범위가 지켜졌는가
- 미확정 정책을 임의로 구현하지 않았는가
- 각 완료 조건에 코드와 검증 증거가 연결되는가

### 4.2 코드 흐름 추적

PR의 코드 읽는 순서를 실제 Diff와 대조한다.

```text
요청 진입
→ Service 유스케이스
→ Domain·Entity 상태 변경
→ Repository·외부 연동
→ 응답·이벤트
→ 테스트
```

PR 가이드가 핵심 경로를 누락하거나 실제 코드와 다르면 발견 사항으로 남긴다.

### 4.3 실패와 정합성 확인

- 검증 순서가 적절한가
- 실패 시 부분 데이터가 남는가
- 상태 전이가 허용된 경로인가
- 금액·수량·참여 인원이 일치하는가
- 인증 정보와 소유권을 서버에서 검증하는가
- 중복·동시 요청 결과가 정의됐는가
- 외부 연동 실패 시 보상 또는 미검증 위험이 기록됐는가

### 4.4 테스트·검증 증거 확인

- 한글 설명형 테스트명인가
- `given → when → then` 구조인가
- 각 테스트의 작성 이유가 완료 조건 또는 위험과 연결되는가
- 실제 테스트 코드가 존재하는가
- 작성과 실행을 구분했는가
- 실행 명령·환경·결과·통과 건수 또는 실패 로그가 있는가
- 상태가 `PASS | FAIL | HOLD | NOT_RUN` 중 하나인가
- 테스트가 보장하는 범위와 한계가 명확한가
- 최신 검증 Commit과 CI 상태가 Head와 일치하는가

단순 테스트 통과만으로 락·트랜잭션·Rollback이 증명됐다고 판단하지 않는다. 실제 조건과 assertion이 이를 확인해야 한다.

### 4.5 Human 답변 확인

담당자 답변을 최신 코드와 비교한다.

확인 대상:

- 기능 목적과 요청·데이터 흐름
- 클래스별 역할
- 상태 변경과 트랜잭션 경계
- 실패 시 결과
- 권한·소유권·동시성
- 테스트의 보장 범위와 한계
- 중요한 선택 이유

답변은 `일치` 또는 `보완 필요`로 설명한다. 점수·합격·불합격·함정 질문·모범 답안은 제공하지 않는다. Human 답변은 테스트와 코드 리뷰를 대체하지 않는다.

## 5. 발견 사항 기준

다음 중 하나가 있을 때만 발견 사항을 남긴다.

- 실제 코드 근거가 있는 결함
- 구체적으로 재현 가능한 실패 시나리오
- 요구사항 누락 또는 계약 충돌
- 검증에 필요한 증거 부족
- 정책·권한·정합성 위험

다음은 기본적으로 발견 사항으로 만들지 않는다.

- 개인 취향 수준의 스타일 의견
- Issue 밖 미래 확장
- 근거 없는 추측
- 사용하지 않는 프레임워크 내부 이론
- 이미 다른 항목에서 다룬 중복 지적
- 이번 변경과 무관한 리팩터링 제안

## 6. 심각도

- **BLOCKER**: 금액·데이터·보안·정합성 또는 핵심 계약을 훼손할 위험
- **MAJOR**: 요구사항 실패, 높은 확률의 런타임 오류, 필수 실패 처리 누락
- **MINOR**: 범위가 제한된 유지보수성·명확성·테스트 공백
- **QUESTION**: 결함으로 단정할 수 없고 계약 확인이 필요한 사항

한 발견 사항에는 하나의 판단만 담는다.

## 7. 발견 사항 형식

```markdown
### [MAJOR] 구체적인 제목

- 위치: `path` `Class#method`
- 확인된 사실:
- 실패 시나리오:
- 영향:
- 최소 수정 방향:
- 확인할 검증:
```

확인된 사실과 추론을 구분한다. 전체 구현을 다시 작성하거나 과도한 칭찬 문구를 붙이지 않는다.

## 8. 최신 Head 재검토

수정 Commit이 올라오면 이전 결과를 그대로 재사용하지 않는다.

1. 현재 Head SHA를 확인한다.
2. 수정된 Diff와 영향 범위를 읽는다.
3. 기존 지적 해결 여부를 확인한다.
4. 관련 테스트·직접 검증·CI가 다시 수행됐는지 확인한다.
5. 새 결함이나 문서 불일치를 확인한다.
6. 해결된 지적은 `해결`, 더 이상 적용되지 않으면 `해당 없음`으로 정리한다.

CI나 로그가 없으면 통과했다고 추정하지 않는다.

## 9. 리뷰 출력 형식

```markdown
## AI 리뷰 요약

- 검토 수준: 기본 | 강화
- 기준 Head:
- 결론: 수정 필요 | 확인 필요 | 특이사항 없음
- 따라간 코드 흐름:
- 확인한 검증:
- 확인하지 못한 범위:

## 발견 사항

### [심각도] 제목
...

## 코드 리뷰 가이드 정합성

- 실제 흐름과 일치 여부:
- 누락된 경로:

## 테스트·검증 증거

- 완료 조건과 테스트 연결:
- 실제 실행 결과:
- 보장 범위와 한계:
- 최신 Commit·CI:

## Human 답변 확인

- 일치:
- 보완 필요:

## 최신 Head 재검토

- 기존 지적 해결 여부:
- 새 발견 사항:
```

발견 사항이 없어도 검토한 범위와 확인하지 못한 위험을 남긴다. 코드를 완벽하다고 표현하지 않는다.

## 10. 경계

리뷰 모드에서는 코드·브랜치·PR 본문을 임의로 수정하거나 Merge하지 않는다.

다음 상황에서는 결론을 추정하지 않고 Human 확인을 요청한다.

- Issue 계약이나 정책이 모호함
- 최신 Diff에 접근할 수 없음
- 실행했다는 테스트에 증거가 없음
- 보안·개인정보·데이터 손실 위험이 있음
- 리뷰 의견이 Issue 범위 확장이나 정책 재결정을 요구함

API Key, 비밀번호, JWT, 운영 환경변수, 실제 개인정보와 식별 가능한 운영 데이터는 리뷰 입력에 포함하지 않는다.
