# ADR 0003: UTC Instant 저장과 Clock 주입 시간 전략

- 상태: `Accepted`
- 작성일: `2026-07-24`
- 관련 Issue: `#18`

## 배경

예약 마감, 결제 만료, 감사 시각처럼 시간에 의존하는 처리와 API 표시 시각은 일관된 기준이 필요하다.

## 문제

시스템 시각을 직접 호출하거나 저장·표시 시간대를 섞으면 시간 의존 로직의 재현과 테스트가 어려워질 수 있다.

## 고려한 대안

- 서비스 로직에서 시스템 시각을 직접 조회한다.
- 절대 시각을 UTC `Instant`로 저장하고 `Clock`을 주입한다.
- 사용자 또는 식당별 시간대를 현재부터 저장 기준으로 사용한다.

## 결정

DB와 Entity의 절대 시각은 UTC 기반 `Instant`로 저장한다. API 응답 시각은 Response DTO에서 `Asia/Seoul` 기준 `OffsetDateTime`으로 명시적으로 변환한다.

서비스 로직은 시스템 시각을 직접 호출하지 않고 주입된 `Clock`을 사용한다. 생성·수정 시각은 공통 `BaseTimeEntity`와 JPA Auditing 기준을 따르며, 테스트에서는 고정 `Clock`으로 시간 의존 로직을 재현한다.

## 선택 이유

저장 기준을 하나로 두고 표시 변환을 응답 계층에 분리하면 시간 계산과 테스트 기준을 명확히 할 수 있다.

## 장점

- 절대 시각 저장 기준이 일관된다.
- 시간 의존 테스트를 고정 시각으로 재현할 수 있다.
- 생성·수정 시각 처리를 공통 골격으로 연결한다.

## 단점과 위험

- 응답 계층에서 시간대 변환을 빠뜨리면 표시 시각이 기대와 달라질 수 있다.
- 다중 비즈니스 시간대 지원이 필요해지면 현재 기준만으로 부족할 수 있다.

## 검증 방법

고정 `Clock` 주입 시 감사 시각과 시간 의존 결과가 같은 기준 시각을 사용하는지 검증한다. 공통 구현 골격은 [COMMON_SKELETON_GUIDE.md](../COMMON_SKELETON_GUIDE.md)와 [CODE_CONVENTION.md](../CODE_CONVENTION.md)를 따른다.

## 재검토 조건

- 서비스가 여러 비즈니스 시간대를 지원할 때
- 시간대가 사용자·식당별 설정으로 바뀔 때
- 저장소 또는 외부 시스템이 다른 시간 표현을 강제할 때
