# ADR 0004: Java 17 프로젝트 기준

- 상태: `Accepted`
- 작성일: `2026-07-24`
- 관련 Issue: `#18`

## 배경

프로젝트의 개발·빌드 기준 Java 버전은 팀 개발 환경과 의존성 호환성, 이후 전환 비용에 영향을 준다.

## 문제

프로젝트 기준 Java 버전이 명확하지 않으면 로컬 개발, CI, 배포 런타임의 호환성 판단이 흔들릴 수 있다.

## 고려한 대안

- 현재 Java Toolchain 17을 유지한다.
- 현재 개발 기간 중 Java 21 이상으로 전환한다.
- Java 버전을 별도 기준 없이 각 환경에 맡긴다.

## 결정

현재 `build.gradle`의 Java Toolchain 17을 프로젝트 개발·빌드 기준으로 유지한다. 팀 개발 환경, 현재 코드와 의존성의 호환성, 프로젝트 기간 중 전환 비용을 함께 고려한다.

Java 21을 금지하거나 영구 배제하지 않으며, 버전 전환은 별도 검증과 새 ADR을 통해 재결정한다.

## 선택 이유

현재 설정과 개발 환경의 기준을 유지해 진행 중인 프로젝트에서 불필요한 전환 위험을 피한다.

## 장점

- 개발·빌드 기준을 `build.gradle`과 일치시킨다.
- 현재 코드와 의존성의 호환성 범위를 유지한다.
- 향후 Java 21 이상 전환 가능성을 닫지 않는다.

## 단점과 위험

- Java 21 이상의 기능이나 성능 이점을 즉시 활용하지 않는다.
- 전환 필요성이 생기면 CI·Docker·배포 런타임을 함께 검증해야 한다.

## 검증 방법

`build.gradle`의 Java Toolchain이 17인지 확인하고, 개발·빌드 환경이 해당 기준을 사용할 수 있는지 검증한다.

## 재검토 조건

- 핵심 의존성이 Java 21 이상을 요구할 때
- Java 21 기능이나 성능 이점이 전환 비용보다 명확히 클 때
- CI·Docker·배포 런타임을 함께 전환할 준비가 완료될 때
