# BobFull Issue 제목 규칙

## 1. 기본 형식

모든 GitHub Issue 제목은 다음 형식을 사용한다.

```text
[버전][유형] 작업 내용
```

예시:

```text
[v1][feat] 회원·인증 및 공통 개발 골격 구성
[v1][test] 예약 동시성 테스트 추가
[v2][fix] 참여자 취소 후 예약 상태 계산 오류 수정
[v3][perf] 예약 가능 좌석 조회 성능 개선
```

## 2. 버전

버전은 `docs/PROJECT_CONTEXT.md`에 확정된 범위를 따른다.

| 버전 | 기준 |
|---|---|
| `v1` | 핵심 예약 거래 흐름과 1차 배포 |
| `v2` | 취소·환불·정산·채팅 등 확장 기능 |
| `v3` | 성능·운영·확장 지역과 고도화 |

작업이 어느 버전에 속하는지 확정되지 않았으면 임의로 붙이지 않고 Human에게 확인한다.

## 3. 유형

Issue 유형은 다음 7개만 사용한다.

| 유형 | 사용 기준 |
|---|---|
| `feat` | 새로운 기능 또는 다른 기능이 의존할 기반 추가 |
| `fix` | 잘못된 동작이나 결함 수정 |
| `test` | 기능 변경 없이 테스트 추가·보강 |
| `docs` | 문서·다이어그램·템플릿만 변경 |
| `refactor` | 동작을 유지하며 내부 구조 개선 |
| `perf` | 성능 개선과 측정 |
| `chore` | 빌드·설정·개발 환경·관리 작업 |

## 4. 유형 선택 기준

- 사용자나 다른 도메인이 사용할 동작·계약이 생기면 `feat`다.
- 코드 결함을 고치면 `fix`다.
- 테스트만 추가하면 `test`다.
- 문서만 바꾸면 `docs`다.
- 외부 동작은 유지하고 내부 구조만 바꾸면 `refactor`다.
- 성능 개선이 목적이고 측정 근거가 있으면 `perf`다.
- 기능이 아닌 설정·환경·단순 관리 작업이면 `chore`다.

`chore`를 애매한 작업의 기본값으로 사용하지 않는다.

## 5. 작성 규칙

- 버전과 유형은 영어 소문자로 작성한다.
- 작업 내용은 한국어로 간결하게 작성한다.
- 제목만 보고 작업 결과를 알 수 있게 작성한다.
- 담당자 이름을 제목에 넣지 않는다.
- 여러 기능을 하나의 제목에 나열하지 않는다.
- 폐기된 정책이나 구현하지 않는 기능명을 사용하지 않는다.

좋은 예:

```text
[v1][feat] 예약 생성 및 좌석 차감 구현
[v1][test] 예약 동시 요청 시 좌석 정합성 검증
[v2][docs] 채팅 접근 정책 문서화
```

나쁜 예:

```text
회원 작업
[v1](test) 테스트 추가
[feat] 이것저것 수정
[v1][chore] 예약 기능 구현
```

## 6. Issue·PR·Commit 구분

Issue 제목은 버전과 작업 유형을 함께 표시한다.

```text
[v1][feat] 회원가입 구현
```

PR과 Commit 제목은 `docs/GITHUB_RULES.md`의 Conventional Commit 형식을 따른다.

```text
feat: 회원가입 구현
```

Issue 유형과 PR·Commit 타입은 가능한 한 일치시킨다.

## 7. AI Issue 작성 규칙

AI가 Issue 초안을 만들 때 다음을 확인한다.

1. `docs/PROJECT_CONTEXT.md`에서 버전 범위를 확인한다.
2. 실제 작업 성격에 맞는 유형을 선택한다.
3. 제목을 `[버전][유형] 작업 내용`으로 작성한다.
4. 버전이나 유형이 불명확하면 추정하지 않고 Human에게 확인한다.
