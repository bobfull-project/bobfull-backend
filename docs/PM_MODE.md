# BobFull PM 모드

## 1. 목적

PM 모드는 아침 스크럼에서 GitHub 현황을 빠르게 정리하고, 팀이 오늘의 목표와 차단 사항을 확정하기 위한 운영 지원 절차다.

AI는 현황과 계획 초안을 만든다. 팀원은 실제 진행 상황을 확인하고, 팀장은 우선순위·담당·일정을 최종 결정한다.

PM 모드는 구현 AI의 코드 작성 절차를 대체하지 않는다.

## 2. 호출어

### 아침 스크럼

```text
PM 모드 시작
```

### 하루 마감

```text
PM 모드 마감
```

## 3. 확인 기준

PM 모드는 다음 순서로 확인한다.

1. `docs/PROJECT_CONTEXT.md`의 일정·버전·역할
2. 열려 있는 GitHub Issue와 `DRAFT | READY` 상태
3. 열려 있는 PR의 대상 브랜치·Draft 여부·리뷰·CI 상태
4. Issue의 선행·후속 관계와 담당 도메인
5. 사용자가 알려준 회의 결과·오프라인 작업·개인 일정

GitHub에 없는 작업을 완료로 추정하지 않는다. 근거가 없거나 실제 상황 확인이 필요한 내용은 `확인 필요`로 표시한다.

진행률은 근거 없이 백분율로 만들지 않는다. 다음처럼 확인 가능한 상태로 표현한다.

```text
미착수 → 설계 중 → READY → 구현 중 → Draft PR → 리뷰 중 → 완료
```

## 5. 아침 스크럼 진행

PM 모드 출력 후 팀은 다음 순서로 확인한다.

1. AI가 저장소 현황과 당일 계획 초안을 설명한다.
2. 각 담당자가 최근 완료, 오늘 목표, 차단 사항, 필요한 협업을 확인한다.
3. GitHub 현황과 실제 상황이 다르면 즉시 수정한다.
4. 선행 작업과 차단 사항을 먼저 조정한다.
5. 팀장이 오늘의 우선순위와 완료 결과를 확정한다.
6. 필요한 Issue·PR 상태와 설명을 갱신한다.

작업 목표는 `기능 작업하기`가 아니라 확인 가능한 결과로 작성한다.

```text
나쁜 예: 회원·인증 작업하기
좋은 예: MemberRole·인증 사용자 객체·공통 예외 구조 구현 후 Draft PR 생성
```

## 6. 오후 중간 점검

전체 보고서를 다시 만들지 않고 다음만 확인한다.

```text
완료
진행 중
차단
오늘 마감 가능 여부
```

새로운 정책 결정이나 다른 담당자 계약 변경이 필요하면 해당 Issue 작업을 멈추고 팀에 공유한다.

## 7. PM 모드 마감 출력

```markdown
# 오늘 마감

## 완료
-

## 미완료
- 작업:
- 이유:
- 현재 상태:

## 차단 사항
-

## 갱신할 GitHub 항목
- Issue:
- PR:

## 내일 첫 작업
-

## 팀장 결정 필요
-
```

## 8. 운영 원칙

- AI가 담당자·우선순위·일정을 임의로 확정하지 않는다.
- 기존 역할과 승인된 Issue를 기준으로 계획한다.
- `DRAFT` Issue를 구현 중으로 표시하지 않는다.
- Push되지 않은 작업과 개인 일정은 담당자 확인을 받는다.
- 선행 작업과 일반 작업을 구분한다.
- 하루에 끝낼 수 있는 결과를 중심으로 배정한다.
- 장기 계획보다 오늘의 완료 조건과 인계 결과를 우선한다.
- 구현 세부 절차는 `docs/AI_WORKFLOW.md`를 따른다.

## 9. 사용 시점

- 매일 아침 스크럼 시작 시 1회
- 필요 시 오후 중간 점검
- 하루 작업 종료 전 1회

PM 모드의 목적은 보고서를 늘리는 것이 아니라, 팀이 같은 현황을 보고 빠르게 실행을 시작하도록 돕는 것이다.
