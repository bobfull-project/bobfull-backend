# 밥풀(BobFull) 제품 요구사항 문서

## 1. 서비스 한 줄 정의

밥풀은 제주에서 혼자 여행하는 사용자가 평일 저녁 식당의 합석 회차를 선택해 함께 예약할 수 있도록 돕는 서비스다.

## 2. 해결하려는 문제

- 2인 이상 주문 또는 테이블 단위 운영으로 1인 고객이 이용하기 어려운 문제
- 함께 식사할 사람을 직접 구해야 하는 불편
- 합석 테이블의 빈 좌석과 낮은 좌석 활용도
- 취소와 노쇼로 인한 식당 손실

## 3. 초기 타깃

- 제주 지역
- 평일 저녁
- 제주를 혼자 여행하는 사용자

이 타깃은 현재 제품 포지셔닝이며, 제주 외 지역을 차단하는 시스템 정책을 뜻하지 않는다.

## 4. 핵심 사용자 흐름

1. OWNER가 식당, 합석 테이블, 예약 가능한 회차를 관리한다.
2. MEMBER가 식당과 회차를 조회하고 본인을 포함한 인원 단위로 예약 결제를 준비한다.
3. PortOne 결제 검증 또는 웹훅 검증이 성공하면 예약 또는 참여 결과를 반영한다.
4. 모집 상태가 열려 있으면 다른 MEMBER가 기존 예약에 참여한다.
5. 모집 마감, 취소, 환불, 노쇼 결과를 예약과 지급 예정 금액 조회에 반영한다.

## 5. MVP 범위

MVP는 회원·인증, 식당·테이블·회차 관리와 사용자 조회, 예약 가능 여부와 결제 준비, 예약·참여 집계 조회, 모집 마감, PortOne 결제 완료 검증·웹훅, 결제·환불 조회, 지급 예정 금액 조회를 포함한다.

## 6. V1·V2·V3 요약

| 버전 | 범위 |
|---|---|
| V1 | MVP 기능, 예약·결제·환불 조회와 지급 예정 금액 |
| V2 | 참여 취소, 노쇼, 채팅, 관리자 조회와 운영 정책 |
| V3 | 실패 결제·환불 재처리, 정산 재집계, 모니터링·성능·배포 고도화 |

## 7. 제외 범위

- 식사대금 결제, POS 연동, 실제 계좌 송금
- 채팅의 읽음 처리, 이미지·파일, 메시지 수정·삭제, 신고·차단
- 현재 확정 범위 밖의 Redis Pub/Sub와 Kafka 기반 채팅 처리

## 8. 현재 제품 결정 요약

- 역할은 MEMBER, OWNER, ADMIN으로 구분한다.
- 예약은 사용자별 `partySize` 단위로 관리하고 예약금 결제는 PortOne으로 처리한다.
- 예약 상태와 모집 상태를 구분하며, 상세 상태 전이와 취소·환불·노쇼 규칙은 별도 기준 문서를 따른다.
- API, 데이터 모델, 운영 정책의 상세 계약은 이 문서에서 다시 정의하지 않는다.
