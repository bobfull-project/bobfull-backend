# 밥풀(BobFull) 프로젝트 확정 컨텍스트

> 기준일: 2026-07-23 최신 초기 기획 반영  
> 이 문서는 저장소에서 사용하는 프로젝트 정책의 단일 기준 문서다.  
> 이 문서와 이전 문서·대화가 충돌하면 이 문서를 우선한다.  
> 최신 기획에 없는 구현 세부사항은 임의로 확정하지 않는다.

---

## 1. 프로젝트 개요

**밥풀**은 혼자 이용하기 어려운 식당에서 1인 사용자 여러 명을 연결해 함께 예약할 수 있도록 지원하는 합석 예약 서비스다.

사장님은 식당의 합석 테이블과 예약 가능 시간대를 등록한다. 합석 테이블의 정원은 `2인·4인·6인·8인` 중 하나로 설정한다.

최초 예약자와 추가 참여자는 각각 본인을 포함한 `partySize`를 입력하고 신청 인원수만큼 PortOne으로 예약금을 결제한다. 현재 참여 인원은 결제 완료 참여자들의 `partySize` 합계로 계산한다.

예약은 테이블별 확정 기준 인원에 도달하면 `CONFIRMED`가 된다. `CONFIRMED` 이후에도 모집 상태가 `OPEN`이고 잔여 정원이 있으면 추가 참여할 수 있다.

## 2. 문제 정의

- 혼자 이용하기 어려운 식당의 접근성 문제
- 함께 식사할 사람을 직접 구해야 하는 불편
- 1인 고객 수용 시 발생하는 빈 좌석 문제
- 예약 취소와 노쇼로 인한 식당 손실
- 예약·결제·환불·노쇼·지급 예정 예약금을 개별적으로 관리해야 하는 문제

## 3. 사용자와 권한

### 일반 사용자

- 예약 가능한 식당·시간·테이블을 조회한다.
- 최초 예약자로 새로운 합석 예약을 생성한다.
- 추가 참여 가능한 예약에 참여한다.
- 본인을 포함한 `partySize`를 입력한다.
- 본인이 신청한 인원수만큼 예약금을 결제한다.
- 현재 참여 인원, 남은 참여 가능 인원, 예약 상태와 모집 상태를 조회한다.
- 정책에 따라 본인 참여를 취소하고 환불 결과를 확인한다.

### 최초 예약자

- 결제 성공 후 예약을 생성하고 모집을 시작한다.
- 예약이 `CONFIRMED`가 된 뒤 추가 모집을 유지하거나 직접 마감할 수 있다.
- `RECRUITING` 상태에서는 확정 기준 충족 전이므로 임의 마감할 수 없다.
- 직접 마감한 모집은 다시 열 수 없다.

### 추가 참여자

- 모집 상태가 `OPEN`인 예약에 남은 참여 가능 인원 범위에서 참여한다.
- 결제 성공 후에만 참여자로 등록된다.
- 동일 예약에 중복 참여할 수 없다.

### 식당 사장님

- 본인 식당과 합석 테이블을 등록·수정·조회한다.
- 테이블 정원을 `2인·4인·6인·8인` 중 하나로 설정한다.
- 테이블별 예약 가능 시간대를 등록한다.
- 예약별 현재 참여 인원과 예약·모집 상태를 조회한다.
- 식사 종료 후 본인 식당 예약의 참여자를 노쇼 처리하거나 해제한다.
- 지급 예정 예약금을 조회한다.
- 다른 사장님의 식당과 예약은 관리할 수 없다.

### 관리자

- 회원·식당·예약·결제·환불·노쇼 현황을 조회한다.
- 시스템 오류와 비정상 처리 내역을 추적한다.
- 식당별 예약 성사율과 사용자별 노쇼율 등 운영 지표를 조회한다.

## 4. 핵심 도메인 구조

```text
사장님
└─ 식당
   ├─ 합석 테이블
   │  ├─ 테이블명
   │  └─ 정원: 2인 / 4인 / 6인 / 8인
   └─ 예약 가능 회차
      ├─ 예약 날짜
      ├─ 예약 시작 시간
      └─ 대상 테이블

예약 가능 회차
└─ 합석 예약
   ├─ 최초 예약자
   ├─ 추가 참여자 여러 명
   ├─ 참여자별 partySize
   ├─ 현재 참여 인원
   ├─ 예약 상태
   └─ 모집 상태
```

## 5. 테이블·회차·좌석 정책

- 좌석 재고 기준은 `합석 테이블 + 예약 시작 시간`이다.
- 동일한 테이블과 시간에는 하나의 합석 예약만 생성할 수 있다.
- 테이블 정원은 `2인·4인·6인·8인`만 사용한다.
- 결제 준비 시 신청한 좌석을 10분간 임시 선점한다.
- PortOne 결제 검증 성공 후에만 최종 예약·참여와 현재 참여 인원을 반영한다.
- 결제 실패 또는 10분 만료 시 임시 선점을 해제한다.
- 예약 전체가 취소되면 테이블·시간을 다시 예약 가능 상태로 복구한다.

```text
현재 참여 인원
= 결제 완료 참여자별 partySize 합계

예약 남은 참여 가능 인원
= 테이블 정원
- 현재 참여 인원

결제 금액
= partySize
× 1인당 예약금
```

## 6. 예약 생성과 추가 참여

### 최초 예약 생성

```text
식당·날짜·시간·테이블 선택
→ partySize 입력
→ 좌석 10분 임시 선점
→ Payment READY 생성과 paymentId 발급
→ PortOne 예약금 결제
→ 서버가 PortOne 결제 상태·금액·통화 검증
→ 검증 성공 시 Payment PAID
→ 합석 예약 생성
→ 최초 참여자 RESERVED 등록
→ 현재 참여 인원에 partySize 반영
→ 확정 기준 미달이면 RECRUITING
→ 확정 기준 이상이면 CONFIRMED
→ 모집 상태 OPEN
→ 결제 실패·만료 시 Payment FAILED와 좌석 해제
```

- `partySize`는 1 이상이어야 한다.
- `partySize`는 테이블 정원을 초과할 수 없다.
- 결제 실패·만료 시 예약·참여자·현재 참여 인원을 반영하지 않고 임시 선점을 해제한다.

### 추가 참여

다음 조건을 모두 만족할 때 참여할 수 있다.

```text
예약 상태가 RECRUITING 또는 CONFIRMED
AND 모집 상태가 OPEN
AND 현재 시각이 식사 시작 2시간 전보다 이전
AND 현재 참여 인원 < 테이블 정원
AND partySize <= 남은 참여 가능 인원
```

```text
추가 참여 요청
→ 동일 예약 중복 참여 검증
→ 남은 참여 가능 인원 검증
→ 좌석 10분 임시 선점
→ Payment READY 생성과 paymentId 발급
→ PortOne 예약금 결제
→ 서버 검증 성공 시 Payment PAID
→ 참여자 RESERVED 등록
→ 현재 참여 인원에 partySize 반영
→ 예약 상태와 모집 상태 재계산
→ 결제 실패·만료 시 Payment FAILED와 좌석 해제
```

## 7. 예약 확정 기준

| 테이블 정원 | `RECRUITING` 범위 | `CONFIRMED` 전환 인원 |
|---:|---:|---:|
| 2인 | 1명 | 2명 |
| 4인 | 1~2명 | 3명 |
| 6인 | 1~4명 | 5명 |
| 8인 | 1~6명 | 7명 |

- 상태 판단 기준은 참여 사용자 수가 아니라 결제 완료된 전체 신청 인원수의 합이다.
- `CONFIRMED`는 식사 진행 확정이며 추가 모집 종료를 뜻하지 않는다.
- `CONFIRMED + OPEN`이고 잔여 정원이 있으면 계속 참여할 수 있다.

## 8. 모집 상태 정책

```text
OPEN    추가 참여 가능
CLOSED  추가 참여 불가능
```

모집 상태가 `CLOSED`가 되는 조건:

- `CONFIRMED` 상태에서 최초 예약자가 직접 마감
- 현재 참여 인원이 테이블 정원에 도달
- 식사 시작 2시간 전 도달

추가 규칙:

- `RECRUITING` 상태에서는 최초 예약자가 모집을 직접 마감할 수 없다.
- 최초 예약자가 직접 마감한 모집은 다시 열 수 없다.
- 모집 마감 시 예약이 이미 `CONFIRMED`이면 예약 상태는 유지한다.
- 모집 마감 시 확정 기준 미달이면 예약 전체를 `CANCELLED`로 변경하고 전액 환불한다.

## 9. 상태 정책

### 예약 상태

```text
RECRUITING  현재 참여 인원이 테이블별 확정 기준 미만
CONFIRMED   현재 참여 인원이 테이블별 확정 기준 이상
CANCELLED   모집 실패·취소·식당·시스템 귀책으로 예약 전체 취소
CLOSED      식사 종료 시간이 지난 예약
```

### 모집 상태

```text
OPEN
CLOSED
```

### 참여자 상태

```text
RESERVED
NO_SHOW
CANCELLED
```

- `RESERVED`: 결제 완료 후 신청 인원 전체가 예약에 참여 중
- `NO_SHOW`: 식사 종료 후 사장님이 해당 참여자를 노쇼 처리
- `CANCELLED`: 사용자가 본인의 참여를 취소
- 정상 방문 참여자는 별도 `VISITED` 상태로 변경하지 않고 `RESERVED`를 유지한다.
- 노쇼 해제 시 `NO_SHOW → RESERVED`로 복구한다.

### 결제 상태

```text
READY      결제 준비와 좌석 임시 선점 완료
PAID       PortOne 결제 검증 성공
FAILED     결제 실패 또는 임시 선점 만료
CANCELLED  환불 완료로 결제 취소 확정
```

결제 상태는 참여자별 결제 건에 적용한다.

### 환불 상태

```text
REQUESTED
PROCESSING
COMPLETED
FAILED
```

- 환불 완료 시 `RefundStatus.COMPLETED`와 `PaymentStatus.CANCELLED`를 함께 반영한다.
- v1에서는 한 사용자의 `partySize` 결제 전체만 환불하며 부분 환불은 지원하지 않는다.

## 10. PortOne 결제 정책

- 고객사가 생성한 `paymentId`를 결제 식별자로 사용한다.
- `paymentOrderId`, `paymentKey`를 서비스 API 계약으로 사용하지 않는다.
- 프론트 결제창 종료 후 서버는 PortOne 결제 단건 조회로 상태·금액·통화를 검증한다.
- PortOne 웹훅도 동일한 결제 정보를 재조회하고 검증한다.
- 결제 완료 검증 API와 웹훅이 동시에 실행돼도 예약·참여·결제 상태는 한 번만 반영한다.
- 사용자가 결제 취소 API를 직접 호출하지 않는다. 예약 취소·모집 실패·식당 귀책 흐름에서 서버가 환불을 요청한다.

## 11. N명 예약 정책 — v1

- 최초 예약자와 추가 참여자는 각각 본인을 포함한 1명 이상의 인원을 신청한다.
- 신청 가능한 최대 인원은 해당 시점의 남은 참여 가능 인원이다.
- 한 사용자의 신청 인원을 하나의 예약 참여 단위로 관리한다.
- 부분 인원 변경을 지원하지 않는다.
- 부분 취소를 지원하지 않는다.
- 부분 노쇼 처리를 지원하지 않는다.
- 취소 또는 노쇼 처리 시 해당 사용자의 신청 인원 전체에 동일하게 적용한다.

## 12. 취소·환불 정책

### 식사 시작 2시간 전까지

- 사용자는 본인의 참여 전체를 취소할 수 있다.
- 취소한 사용자가 결제한 예약금은 전액 환불한다.
- 참여자 상태를 `CANCELLED`로 변경하고 `partySize`를 현재 참여 인원에서 차감한다.

최초 예약자 취소:

- 예약 전체를 `CANCELLED`로 변경한다.
- 나머지 참여자 전원에게 예약금을 전액 환불한다.
- 해당 테이블·시간을 다시 예약 가능 상태로 복구한다.

추가 참여자 취소:

- 취소 후 확정 기준 이상이면 `CONFIRMED`를 유지한다.
- 확정 기준 미만이고 모집 상태가 `OPEN`이면 `RECRUITING`으로 변경하고 재모집한다.
- 확정 기준 미만이고 최초 예약자가 모집을 직접 마감한 상태라면 예약 전체를 취소하고 귀책 없는 참여자를 전액 환불한다.
- 모집 마감까지 확정 기준을 다시 충족하지 못하면 전체 취소하고 남은 참여자를 전액 환불한다.

### 식사 시작 2시간 이후

- 추가 참여자를 모집하지 않는다.
- 사용자는 개인 사유로 본인의 참여 전체를 취소할 수 있지만 본인 예약금은 환불하지 않는다.
- 미환불 예약금은 지급 예정 예약금에 포함한다.
- 취소 후 남은 참여 인원이 확정 기준 미만이 되어 식사가 진행되지 못하면 예약 전체를 취소한다.
- 귀책 없는 나머지 참여자에게는 예약금을 전액 환불한다.

### 식당·시스템 귀책

- 예약 전체를 `CANCELLED`로 변경한다.
- 참여자 전원에게 예약금을 전액 환불한다.
- 테이블·시간을 다시 예약 가능 상태로 복구한다.
- 사용자 노쇼율에는 반영하지 않는다.

## 13. 노쇼 정책

- 방문 코드는 사용하지 않는다.
- 식사 종료 후 사장님이 본인 식당의 예약 참여자 목록을 조회한다.
- 사장님은 미방문 참여자를 `NO_SHOW`로 처리한다.
- 한 참여자의 `partySize` 전체를 하나의 묶음으로 처리한다.
- 잘못 처리한 경우 사장님은 노쇼 처리를 해제할 수 있다.
- 노쇼 해제 시 참여자 상태를 `RESERVED`로 복구한다.
- `NO_SHOW` 참여자의 예약금은 환불하지 않고 지급 예정 예약금에 포함한다.
- 식당·시스템 귀책 취소 건은 노쇼 처리할 수 없다.
- 예약 시간 이전에는 노쇼 처리할 수 없다.
- 노쇼 처리와 해제 이력에는 처리한 사장님과 처리 시각을 저장한다.

```text
노쇼율
= NO_SHOW 인원
÷ 예약 종료 참여 인원
```

## 14. 식사 종료와 지급 예정 예약금

- 식사 종료 시간이 지나면 예약을 `CLOSED`로 변경한다.
- `CLOSED` 상태에서도 사장님은 참여자별 노쇼 처리와 해제를 할 수 있다.
- 실제 계좌 송금 기능은 구현하지 않는다.

```text
지급 예정 예약금
= 전체 예약금 결제 완료 금액
- 전체 예약금 환불 완료 금액
```

포함:

- 정상 이용자의 예약금
- 노쇼 참여자의 예약금
- 식사 시작 2시간 이후 취소하여 환불되지 않은 예약금

제외:

- 모집 실패 환불
- 식당 귀책 환불
- 시스템 귀책 환불
- 다른 참여자의 취소로 식사가 무산되어 귀책 없는 참여자에게 환불된 금액

## 15. 동시성·정합성 요구사항

- 동일 테이블과 시간에 하나의 합석 예약만 생성할 수 있다.
- 동일 사용자는 같은 예약에 중복 참여할 수 없다.
- 여러 사용자가 동시에 N명분 참여를 요청해도 합산 인원이 테이블 정원을 초과하면 안 된다.
- 결제 성공 참여자만 현재 참여 인원에 반영한다.
- 예약 취소·환불·참여 인원·예약 상태·모집 상태·회차 복구 결과가 서로 일치해야 한다.
- 구체적인 락 전략은 기능 Issue에서 결정하고 동시성 테스트로 검증한다.

## 19. 폐기된 이전 정책

다음 내용은 더 이상 기준으로 사용하지 않는다.

- 모든 사용자가 본인 1명분만 예약·결제
- 테이블 정원 `2인·4인·8인`만 사용
- 최소 성사 인원 2명 고정
- 참여 사용자 수로 예약 상태 판단
- 예약 상태만으로 추가 모집 가능 여부 판단
- `VISITED` 참여자 상태
- `CONFIRMED = 모집 종료`
- 방문 코드와 사용자 직접 체크인

## 20. API 공통 계약

- Base URL은 `/api`다.
- 기능 단계는 `V1·V2·V3`로 관리하지만 URL에는 버전을 넣지 않는다.
- 일반 사용자 역할명은 코드와 동일하게 `MEMBER`를 사용한다.
- 성공 응답은 `success`, `message`, `data`를 사용하고 성공 `code`는 생략한다.
- 실패 응답은 `success`, `message`, `code`를 사용하고 `data`는 생략한다.
- 역할 부족은 공통 `ACCESS_DENIED`, 타인 리소스 접근은 도메인별 접근 거부 코드로 처리한다.
