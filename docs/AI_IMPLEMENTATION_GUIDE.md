# BobFull 담당자 AI 실행 가이드

## 1. 실행 명령

최초 온보딩 이후 담당자 AI에는 다음 명령만 사용한다.

```text
Issue #번호 구현하라
```

담당자 AI는 명령을 받을 때마다 GitHub Issue, 연결된 PR, 최신 Head, PR 리뷰·댓글, 현재 브랜치와 작업 트리를 다시 읽고 현재 상태의 다음 단계부터 재개한다.

## 2. 상태별 자동 동작

### A. Issue 분석과 Human 질문

조건:

- Issue에 Human 답변이 없음

동작:

1. 확정 문서·Issue·코드·테스트를 분석한다.
2. 충돌과 미결정 사항을 확인한다.
3. 구현에 필요한 질문을 Issue의 `Human 이해도`에 작성한다.
4. 상태를 `HUMAN_ANSWER_REQUIRED`로 남긴다.
5. 구현하지 않고 중단한다.

### B. Human 답변 검증과 최종 Issue 작성

조건:

- Human 답변이 작성됨
- Issue가 아직 `AI_FINALIZED`가 아님

동작:

1. Human 답변을 문서·코드와 대조한다.
2. Human 답변을 덮어쓰거나 대리 작성하지 않는다.
3. 정확한 답변에는 필요한 맥락을 `AI 보완 설명`으로 추가한다.
4. 부족하거나 잘못된 답변에는 `보완 필요`와 근거를 작성한다.
5. 답변을 반영해 범위·제외 범위·정책·흐름·완료 조건·검증 계획을 최종 재작성한다.
6. 충돌이 남으면 `HOLD`, 구현 가능하면 `AI_FINALIZED`로 남긴다.
7. Human이 최종 Issue를 읽을 때까지 구현하지 않는다.

### C. Human 구현 승인 기록과 구현·Draft PR

조건:

- Issue 상태가 `AI_FINALIZED`
- Human이 최종 계약을 확인한 뒤 다시 `Issue #번호 구현하라`를 입력함

해당 명령을 Human의 구현 승인 신호이자 구현 지시로 해석한다.

구현 전 승인 기록:

1. 현재 Issue가 `AI_FINALIZED`이며 `HOLD` 또는 미결정 사항이 없는지 확인한다.
2. Issue의 `구현 승인 기록`에 다음을 작성한다.
   - 승인 주체: 명령을 입력한 Human 담당자
   - 승인 신호: `Issue #번호 구현하라`
   - 기록 주체: 담당자 AI
   - 승인 대상: 현재 `AI 최종 Issue 검증`의 최종 계약
   - 승인 시각: 실제 명령 확인 시각
   - 승인 상태: `HUMAN_APPROVED`
3. Issue 상태를 `HUMAN_APPROVED`로 변경한다.
4. Human에게 별도 승인 문구나 체크박스 작성을 요구하지 않는다.
5. 승인 기록을 남긴 같은 실행에서 구현을 계속한다.

구현 동작:

1. `docs/GITHUB_RULES.md`에 맞는 기능 브랜치를 확인한다.
2. develop 또는 main이고 작업 트리가 깨끗하면 기능 브랜치를 생성한다.
3. 미관련 변경이 있으면 중단한다.
4. 승인 기록이 연결된 최종 Issue 범위의 최소 변경 계획을 세운다.
5. 코드·테스트·필요 문서를 구현한다.
6. 테스트와 직접 검증을 실제로 실행한다.
7. 전체 실제 Diff를 자체 검토한다.
8. PR에 담당자 Human 이해도 질문과 Human 리뷰 양식을 포함한다.
9. Issue 관련 변경만 Commit·Push한다.
10. develop 대상 Draft PR을 생성한다.
11. PR 상태를 `HUMAN_ANSWER_REQUIRED`로 남기고 중단한다.

승인 이후 최종 계약이 변경되면 기존 승인을 무효화한다. 담당자 AI는 구현을 중단하고 Issue 상태를 `AI_FINALIZED` 또는 `HOLD`로 되돌린 뒤 새 Human 승인 명령을 기다린다.

### D. 연결된 PR 검토와 리뷰 반영

조건:

- 대상 Issue에 연결된 PR이 존재함

다른 팀원의 AI 리뷰는 이 단계의 시작 조건이 아니다. PR에 리뷰·댓글이 없어도 담당자 AI는 최신 Head 자체 검토와 Human 답변 검증을 수행한다.

동작:

1. 최신 Head SHA와 실제 Diff를 확인한다.
2. 최종 Issue 계약, `HUMAN_APPROVED` 기록과 승인 이후 변경 여부를 확인한다.
3. PR 본문, 테스트·build·직접 검증·CI 증거를 확인한다.
4. 담당자 Human 답변을 실제 코드와 대조한다.
5. 각 답변 아래 `AI 답변 검토`와 `AI 보완 설명`을 작성한다.
6. Human 리뷰의 리뷰어·기준 Head SHA·리뷰 시각·재확인 상태를 확인한다.
7. PR에 등록된 모든 리뷰·댓글을 작성 주체와 관계없이 읽는다.
8. 실제 코드 근거가 있는 범위 안 결함만 수정한다.
9. 단순 질문·설명 요청은 확인 결과로 정리한다.
10. 정책·API·DB·권한·트랜잭션 재결정은 Human에게 보고한다.
11. 영향받는 테스트·직접 검증·전체 build를 다시 실행한다.
12. 실제 Diff, PR 본문, 검증 증거와 최신 Head를 갱신한다.
13. 남은 BLOCKER가 없고 필수 Human 입력이 확인되면 `FINAL_HUMAN_REVIEW`로 보고한다.

## 3. 구현 원칙

- 한 번에 하나의 Issue만 처리한다.
- `HUMAN_APPROVED` 기록과 연결된 최신 최종 Issue 계약을 기준으로 구현한다.
- 존재하지 않는 코드·설정·정책을 추정하지 않는다.
- 기존 패키지 구조와 코드 컨벤션을 유지한다.
- 범위 밖 기능·리팩터링·새 기술을 추가하지 않는다.
- 인증 사용자 ID를 클라이언트 입력값으로 신뢰하지 않는다.
- 상태·금액·수량·권한은 실패 후 결과까지 확인한다.
- 일반 컴파일 오류와 테스트 오류는 범위 안에서 수정한다.
- 정책·구조 재결정이 필요한 오류는 Human에게 보고한다.
- PR 리뷰·댓글은 작성 주체가 아니라 실제 근거로 판단한다.
- 특정 팀원이나 다른 AI의 리뷰 존재 여부를 완료 조건으로 사용하지 않는다.

## 4. 테스트와 검증

`docs/TEST_CONVENTION.md`를 따른다.

- 한글 설명형 테스트명
- `given → when → then`
- 완료 조건과 실제 실패 위험에 연결된 테스트
- 실제 실행 명령과 환경
- `PASS | FAIL | HOLD | NOT_RUN`
- 통과 건수 또는 실패 로그
- 직접 검증 결과
- 보장 범위와 미검증 범위

실행하지 않은 테스트를 `PASS`로 기록하지 않는다. 전체 build가 실패하면 성공 또는 완료로 표현하지 않는다.

## 5. 실제 Diff 자체 검토

Commit 전과 PR 처리 시 다음을 확인한다.

- Issue 범위 밖 변경
- 요청부터 응답·저장까지 실제 흐름
- 입력 검증과 예외 처리
- 권한·소유권
- 트랜잭션·Rollback·부분 성공
- 상태·금액·수량 정합성
- 중복·동시 요청 위험
- 테스트와 완료 조건 연결
- 문서·API·DB 변경 반영
- 비밀정보·개인정보 포함 여부

담당자 AI의 자체 검토는 Human Approve를 대체하지 않는다.

## 6. Human 입력

### 6.1 담당자 Human 이해도

Issue와 PR 모두 최신 계약·Diff·테스트를 기준으로 담당자가 반드시 이해해야 할 질문만 작성한다.

- 작은 변경: 1~2개
- 일반 기능: 3~5개
- 권한·결제·좌석·동시성: 4~6개

담당자 Human은 답변을 직접 작성한다. 담당자 AI는 다음만 작성한다.

```markdown
**AI 답변 검토:** `일치 | 보완 필요 | 미작성`

**AI 보완 설명:**
```

### 6.2 Human 리뷰 양식

PR 템플릿에는 PR 작성자가 아닌 Human 리뷰어가 작성할 다음 양식을 포함한다.

```markdown
## Human 리뷰

- 리뷰어:
- 기준 Head SHA:
- 리뷰 시각:
- 수정 후 재확인: `필요 | 완료 | 해당 없음`

### 내가 이해한 구현 흐름

-

### 이해되지 않거나 추가 설명이 필요한 부분

-

### 문제로 보이거나 다시 확인할 부분

-
```

리뷰어가 여러 명이면 양식을 복사한다. 새 Commit 이후 기준 Head가 달라지면 재확인 상태를 `필요`로 본다.

담당자 AI는 Human 리뷰 원문을 작성하거나 덮어쓰지 않는다. 리뷰 내용은 실제 Diff와 대조해 결함, 설명 필요, Human 결정 필요 항목으로 구분한다.

## 7. PR 리뷰·댓글 분류

PR에 등록된 리뷰·댓글은 워크플로우 필수 단계가 아닌 참고 입력이다.

### 자동 수정 가능한 항목

- Issue 계약 안의 명확한 기능 오류
- 예외 처리·검증·테스트 누락
- PR 설명·Diff 불일치
- 정책 재결정이 필요 없는 BLOCKER·MAJOR·MINOR
- Human 리뷰에서 실제 코드 근거가 확인된 결함

### 자동 수정하지 않는 항목

- 근거를 확인할 수 없는 의견
- 단순 질문·설명 요청
- Issue 범위 확장
- 정책·API·DB·상태·권한·트랜잭션 재결정
- 다른 담당자 계약 변경
- 새로운 라이브러리·인프라 도입

반영 여부와 이유를 PR의 `담당자 AI 검토·수정 기록`에 남긴다.

## 8. Draft PR 기록

PR에는 실제 확인한 내용만 기록한다.

- 관련 Issue와 검토 수준
- Issue의 `HUMAN_APPROVED` 승인 기록
- 주요 변경과 제외 범위
- 코드 확인 순서
- 완료 조건별 구현·검증 증거
- 실제 테스트·build·직접 검증 결과
- 최신 Commit SHA와 CI 상태
- 미검증 범위와 남은 위험
- 담당자 Human 이해도 질문
- Human 리뷰 양식
- 담당자 AI 검토·수정 기록

## 9. 즉시 중단 조건

- 문서·Issue·코드가 충돌함
- Human 답변이 모호하거나 미결정 사항이 남음
- 구현 승인 기록이 없거나 승인 이후 최종 계약이 변경됨
- 정책·API·DB·상태·권한·트랜잭션 재결정이 필요함
- 다른 담당자 계약 변경이 필요함
- 새로운 라이브러리나 인프라 결정이 필요함
- 데이터 손실·정합성·보안 위험이 있음
- 핵심 검증 환경이나 권한이 없음
- Issue 범위 밖 변경이 필요함

이미 존재하는 PR은 확인 가능한 최신 Diff까지 검토하고, 중단 원인과 Human 결정 필요 사항을 보고한다.

## 10. 완료 경계

담당자 AI는 다음까지만 수행한다.

```text
Issue 분석·최종화
→ Human 승인 기록
→ 구현·테스트
→ Commit·Push·Draft PR
→ 최신 Head 자체 검토
→ Human 답변 보완
→ 등록된 리뷰·댓글 판단
→ 범위 안 지적 수정·재검증·Push
```

Approve와 Merge는 Human이 수행한다.
