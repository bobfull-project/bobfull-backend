# BobFull 구현 AI 실행 가이드

## 1. 목적

이 문서는 Codex, Claude Code 등 코드 구현 에이전트가 승인된 GitHub Issue를 구현하고 테스트·검증·Draft PR까지 인계할 때 따르는 실행 기준이다.

구현 AI의 목적은 코드를 많이 생성하는 것이 아니다.

- 승인된 요구사항을 빠짐없이 구현한다.
- Issue 범위 안에서 최소한으로 변경한다.
- 실제 실행 결과와 미검증 범위를 구분한다.
- 담당자가 구현 결과를 이해할 수 있도록 실제 Diff 기반 질문을 남긴다.
- 설계·정책·리뷰 반영과 Merge의 최종 판단은 Human에게 돌려준다.

> 구현 AI는 승인된 계약을 코드와 검증 근거로 변환한다.
> 정책 결정과 결과 책임은 Human에게 있다.

---

## 2. 작업 전 필수 입력

구현 AI는 작업 시작 전에 다음을 확인한다.

1. 대상 GitHub Issue
2. Issue의 구현 시작 승인 여부
3. 구현 범위와 제외 범위
4. 완료 조건
5. 테스트와 직접 검증 계획
6. 관련 코드와 기존 테스트
7. 필요한 프로젝트 문서

확인할 문서:

| 작업 | 문서 |
|---|---|
| 공통 AI 경계 | `AGENTS.md` |
| 전체 AI 협업 흐름 | `docs/AI_WORKFLOW.md` |
| Git 브랜치·커밋·PR | `docs/GITHUB_RULES.md` |
| Java·Spring 구현 | `docs/CODE_CONVENTION.md` |
| PR 형식 | `.github/pull_request_template.md` |

존재하지 않는 문서나 확정되지 않은 정책을 추정해 채우지 않는다.

---

## 3. 작업 시작 조건

다음 조건이 모두 충족된 경우에만 구현한다.

- 대상 Issue가 명확하다.
- Human이 구현 시작을 승인했다.
- 핵심 정책과 완료 조건이 정해졌다.
- API·DB 변경이 필요한 경우 승인 범위가 명시됐다.
- 관련 담당자 영역의 계약이 필요한 경우 검토가 끝났다.

다음 상태에서는 구현하지 않는다.

- 아이디어 또는 초안만 존재함
- 핵심 정책이 미결정임
- Issue와 문서가 충돌함
- API·ERD·DB 구조를 새로 결정해야 함
- 작업 범위가 여러 Issue에 걸쳐 있음

---

## 4. 기본 실행 흐름

```text
Issue 계약 확인
→ 관련 문서·코드·테스트 확인
→ 현재 브랜치와 변경 상태 확인
→ 영향 범위와 최소 변경 계획 작성
→ 구현
→ 테스트와 직접 검증
→ 실제 Diff 자체 검토
→ 검증 결과·미검증 범위 정리
→ Human 이해도 질문 작성
→ Commit·Push
→ Draft PR 작성
```

한 번에 하나의 승인된 Issue만 처리한다.

---

## 5. 구현 계획

구현 전에 짧은 계획을 작성한다.

필수 항목:

- 변경할 주요 파일과 역할
- 핵심 데이터 흐름
- 상태 변경 또는 트랜잭션 경계
- 작성하거나 수정할 테스트
- 직접 검증이 필요한 항목
- 범위 밖이라 변경하지 않을 항목

계획은 실행을 위한 최소 수준으로 작성한다.
이미 승인된 요구사항을 다시 장문의 설계 문서로 만들지 않는다.

---

## 6. 구현 원칙

- Issue 완료 조건을 기준으로 구현한다.
- Issue 범위 밖 기능을 추가하지 않는다.
- 불필요한 리팩터링을 함께 수행하지 않는다.
- 서비스 정책, API와 DB 구조를 임의로 변경하지 않는다.
- 기존 코드가 불편하다는 이유만으로 다른 담당자 영역을 재구성하지 않는다.
- 생성한 코드가 `docs/CODE_CONVENTION.md`를 따르는지 확인한다.
- 인증 사용자의 식별자는 클라이언트 요청값보다 인증 정보를 기준으로 사용한다.
- Entity 상태는 의미 있는 메서드로 변경한다.
- 트랜잭션 안에서 불필요한 외부 호출과 긴 대기를 피한다.
- 코드 작성 전에 현재 구현과 기존 테스트를 먼저 읽는다.

일반적인 구현 오류는 승인된 범위 안에서 자체 수정할 수 있다.
설계나 정책 재결정이 필요한 문제는 자체 판단으로 해결하지 않는다.

---

## 7. 테스트와 직접 검증

### 7.1 테스트 작성

Issue의 완료 조건을 증명하는 테스트를 우선 작성한다.

가능한 범위에서 다음을 검증한다.

- 정상 흐름
- 입력 오류와 실패 흐름
- 상태 경계
- 권한과 소유권
- 트랜잭션과 Rollback
- 중복 요청 또는 동시 요청
- 정원·금액·상태 정합성

모든 테스트 종류를 기계적으로 추가하지 않는다.
실제 계약과 위험에 필요한 테스트만 작성한다.

### 7.2 테스트 실행

실제로 실행한 명령과 결과만 기록한다.

다음을 구분한다.

```text
실행한 명령
성공한 테스트
실패한 테스트
실행하지 못한 테스트
환경 문제로 확인하지 못한 항목
```

테스트를 실행하지 않았으면 통과했다고 작성하지 않는다.

### 7.3 직접 검증

자동 테스트만으로 확인하기 어려운 경우 직접 검증한다.

예시:

- HTTP 요청과 응답
- DB 저장 행과 상태
- 권한이 없는 사용자의 접근 차단
- 동시 요청 이후 최종 참여 인원
- 결제 실패 후 예약·참여자 미생성
- Rollback 이후 데이터 상태

직접 검증을 하지 않았다면 `미검증`으로 기록한다.

---

## 8. 검증 결과 작성 기준

항상 다음을 분리한다.

- 확인된 사실
- 실행한 결과
- 추정
- 테스트가 보장하는 범위
- 테스트가 보장하지 않는 범위
- 미검증 항목
- 남아 있는 위험

다음과 같이 과장하지 않는다.

```text
테스트 성공 ≠ 요구사항 전체 충족
단위 테스트 성공 ≠ 실제 운영 환경 검증
기능 동작 ≠ 동시성 안전
일부 응답 측정 ≠ 성능 개선 증명
```

---

## 9. 실제 Diff 자체 검토

구현과 검증을 마친 뒤 최신 실제 Diff를 다시 읽는다.

확인 항목:

- Issue 범위 밖 변경이 없는가
- 빠진 완료 조건이 없는가
- 불필요한 파일이 변경되지 않았는가
- 디버그 코드와 임시 설정이 남지 않았는가
- 테스트를 삭제하거나 약화하지 않았는가
- 예외·권한·Rollback 처리가 요구사항과 맞는가
- 코드와 문서·PR 설명이 일치하는가
- 실행하지 않은 검증을 성공으로 작성하지 않았는가

계획이나 기억이 아니라 실제 Diff를 기준으로 검토한다.

---

## 10. 구현 후 Human 이해도 질문

구현 AI는 코드와 테스트를 완료하고 실제 Diff를 검토한 뒤 Human 이해도 질문을 작성한다.

질문의 목적은 담당자를 시험하거나 어려운 이론을 요구하는 것이 아니다.
담당자가 자신이 구현한 기능의 목적, 흐름, 핵심 코드, 실패 조건과 검증 범위를 설명할 수 있는지 확인하는 것이 목적이다.

### 10.1 질문 생성 근거

질문은 반드시 다음 중 하나 이상의 실제 근거를 사용한다.

- 대상 Issue의 완료 조건
- 실제 변경된 코드
- 상태 변경 로직
- 트랜잭션 또는 Rollback 경계
- 인증·인가 또는 소유권 검증
- 동시성 처리
- 작성하거나 수정한 테스트
- 직접 검증 결과
- 남아 있는 미검증 위험

예상 구현이나 계획만 보고 질문하지 않는다.
실제 구현과 관련 없는 기술은 질문하지 않는다.

### 10.2 우선 질문 관점

- 이 기능이 해결하는 문제
- 요청부터 응답까지의 핵심 흐름
- 주요 클래스와 메서드의 역할
- Entity와 데이터 상태가 변경되는 과정
- 트랜잭션 범위와 실패 시 결과
- 권한을 어디서 어떻게 확인하는지
- 동시 요청에서 발생 가능한 문제와 대응
- 테스트가 보장하는 내용
- 테스트가 보장하지 못하는 내용
- 이번 구현 방식의 핵심 선택 이유

모든 관점을 반드시 질문하지 않는다.
실제 Diff에서 반드시 알아야 할 핵심만 고른다.

### 10.3 난이도 기준

질문은 신입 백엔드 개발자의 기술면접 수준을 기준으로 한다.

중요 기능이라고 해서 이론 난이도를 높이지 않는다.
대신 실패 조건, 정합성, 권한, Rollback과 검증 범위를 더 확인한다.

다음 질문은 만들지 않는다.

- 이번 구현에서 사용하지 않은 기술
- Spring·JVM·Hibernate·DB의 지나치게 깊은 내부 구현
- 라이브러리 내부 알고리즘 또는 수학적 증명
- 클래스명, 설정값과 명령어 암기
- 실제 Diff와 관계없는 기술 비교
- 같은 내용을 지나치게 잘게 나눈 반복 질문
- 담당자를 곤란하게 만들기 위한 함정 질문

### 10.4 질문 수

- 문서·설정·작은 변경: 1~2개
- 일반 CRUD·기능 구현: 3~5개
- 결제·환불·좌석·권한·동시성·상태 전이: 4~6개

정해진 개수를 채우기 위해 불필요한 질문을 추가하지 않는다.
핵심을 확인할 수 있다면 최소 개수로 작성한다.

### 10.5 좋은 질문 예시

- 이 API 요청이 들어온 뒤 어떤 순서로 데이터를 조회하고 저장하나요?
- Request의 `memberId`를 신뢰하지 않고 인증 객체에서 ID를 가져오는 이유는 무엇인가요?
- 결제 성공 후 예약 저장이 실패하면 어떤 결과가 되어야 하나요?
- 여러 사용자가 마지막 자리에 동시에 참여하면 어떤 문제가 발생할 수 있고, 이번 구현은 이를 어떻게 막나요?
- 예약 상태를 setter가 아니라 상태 변경 메서드로 변경한 이유는 무엇인가요?
- 작성한 테스트는 어떤 흐름을 검증하고 어떤 부분까지는 보장하지 못하나요?

### 10.6 피해야 할 질문 예시

- Spring AOP 프록시 생성 과정을 내부 클래스 구조까지 설명하세요.
- InnoDB의 next-key lock과 gap lock 내부 동작을 상세히 증명하세요.
- Hibernate Dirty Checking 알고리즘 내부 구현을 설명하세요.
- JWT 서명 알고리즘의 수학적 원리를 설명하세요.
- 사용하지 않은 여러 동시성 제어 방식을 모든 관점에서 비교하세요.

### 10.7 구현 AI의 금지 사항

구현 AI는 질문만 작성한다.

다음을 하지 않는다.

- Human 답변 대신 작성
- 모범 답안 또는 보완 답안 작성
- 답변 채점
- PASS·FAIL 판정
- 재제출 요구
- Human 최초 답변 수정 또는 삭제
- 이해도 답변을 이유로 승인되지 않은 코드 수정

Human 답변과 실제 코드의 정합성은 ChatGPT PR 리뷰 단계에서 확인한다.

## 11. Draft PR 작성

Draft PR에는 최소한 다음을 기록한다.

- 연결된 Issue
- 구현 목적과 범위
- 주요 변경 내용
- 제외한 범위
- 실행한 테스트와 결과
- 직접 검증 결과
- 미실행·미검증 항목
- 남아 있는 위험
- 구현 AI가 작성한 Human 이해도 질문

Human 답변 영역은 비워두거나 안내 주석만 남긴다.

```markdown
## Human 이해도

### 구현 AI 질문

1. 질문
2. 질문
3. 질문

### 담당자 답변

<!-- 담당자가 실제 코드와 테스트를 확인한 뒤 직접 작성 -->
```

PR은 처음에는 Draft로 작성한다.

---

## 12. 리뷰 수정

ChatGPT 또는 Human 리뷰가 작성됐다고 자동으로 수정하지 않는다.

Human이 항목별로 다음 중 하나를 결정한 뒤 작업한다.

- `반영`
- `미반영`
- `후속 Issue`

구현 AI는 `반영`으로 승인된 항목만 수정한다.

수정 후:

1. 관련 테스트 재실행
2. 필요한 직접 검증 재수행
3. 실제 Diff 재검토
4. 검증 결과 갱신
5. Commit·Push
6. 최신 Head 재검토 인계

수정 과정에서 새로운 설계 결정이 필요해지면 중단한다.

---

## 13. 중단 조건

다음 상황에서는 구현을 중단하고 Human 판단을 요청한다.

- Issue와 문서 또는 현재 코드가 충돌함
- 서비스 정책이 확정되지 않음
- API·ERD·DB 구조 재결정이 필요함
- 다른 담당자 영역의 계약 변경이 필요함
- 결제·좌석·권한·데이터 손실 위험이 발견됨
- 테스트 삭제 또는 약화가 필요함
- Issue 범위 밖 변경이 필요함
- 다른 작업자의 변경을 덮어쓸 위험이 있음
- 핵심 검증에 필요한 환경이나 권한이 없음

보고 형식:

```text
확인된 문제
확인된 사실
추정 원인
영향 범위
가능한 선택지
권장 방향
필요한 Human 결정
현재까지 수행한 검증
```

확인되지 않은 추정을 사실처럼 보고하지 않는다.

## 14. Git 작업 경계

- Git 작업은 `docs/GITHUB_RULES.md`를 따른다.
- `main`과 `develop`에 직접 Commit·Push하지 않는다.
- 기능 브랜치는 `develop`을 기준으로 생성한다.
- 하나의 브랜치는 하나의 Issue 또는 하나의 기능에 집중한다.
- 기능 PR은 `develop`을 대상으로 한다.
- AI는 Merge하지 않는다.

브랜치·Commit·Push·Draft PR 수행에 필요한 권한이 없다면 가능한 구현과 검증까지 진행하고, 수행하지 못한 Git 작업을 명확히 보고한다.

---

## 15. 완료 인계 조건

다음 조건을 충족한 뒤 Draft PR을 Human에게 인계한다.

- 승인된 Issue 범위를 구현함
- 완료 조건을 코드와 테스트로 확인함
- 실행한 테스트와 결과를 기록함
- 미실행·미검증 항목을 기록함
- 실제 Diff 자체 검토를 완료함
- 범위 밖 변경이 없음을 확인함
- Human 이해도 질문을 작성함
- 질문이 실제 Diff 기반이며 난이도 제한을 지킴
- PR 본문이 실제 변경과 일치함
- Merge하지 않고 Human 검토 상태로 인계함

구현 AI의 완료는 Merge가 아니다.
Human 리뷰와 승인, 최신 Head 확인 이후에만 Merge할 수 있다.
