# BobFull 구현 AI 실행 가이드

## 1. 실행 명령

최초 온보딩 이후 구현 에이전트에는 다음 명령만 사용한다.

```text
Issue #번호 구현하라
```

구현 AI는 명령을 받을 때마다 GitHub Issue, 연결된 PR, 현재 브랜치와 작업 트리를 다시 읽고 현재 상태의 다음 단계부터 재개한다.

## 2. 상태별 자동 동작

### A. Human 답변 대기

조건:

- Issue가 `DRAFT`이거나 Human 착수 답변이 없음

동작:

1. 확정 문서·Issue·코드·테스트를 분석한다.
2. 충돌과 미결정 사항을 확인한다.
3. 구현에 필요한 Human 질문을 Issue에 작성한다.
4. 상태를 `HUMAN_ANSWER_REQUIRED`로 남긴다.
5. 구현하지 않고 중단한다.

### B. 최종 Issue 계약 작성

조건:

- Human 착수 답변이 작성됨
- Issue가 아직 `READY`가 아님

동작:

1. Human 답변을 문서·코드와 검증한다.
2. 답변을 수정하거나 대리 작성하지 않는다.
3. 필요한 설명을 `AI 보완 설명`으로 추가한다.
4. 답변을 반영해 구현 범위·제외 범위·정책·흐름·완료 조건·검증 계획을 최종 정리한다.
5. 충돌이 남으면 `HOLD`, 구현 가능하면 `READY_CANDIDATE`로 남긴다.
6. Human의 최종 READY 확인을 기다린다.

### C. 구현·테스트·Draft PR

조건:

- 실제 GitHub Issue 상태가 `READY`
- READY 체크리스트와 승인자가 기록됨
- 구현을 막는 미결정 사항이 없음

동작:

1. `docs/GITHUB_RULES.md`에 맞는 기능 브랜치를 확인한다.
2. develop 또는 main이고 작업 트리가 깨끗하면 기능 브랜치를 생성한다.
3. 미관련 변경이 있으면 중단한다.
4. Issue 범위의 최소 변경 계획을 세운다.
5. 코드·테스트·필요 문서를 구현한다.
6. 테스트와 직접 검증을 실제로 실행한다.
7. 전체 실제 Diff를 자체 검토한다.
8. 구현 후 Human 이해도 질문을 작성한다.
9. Issue 관련 변경만 Commit·Push한다.
10. develop 대상 Draft PR을 생성한다.
11. Human의 PR 확인과 답변을 기다린다.

### D. 1차 리뷰 반영

조건:

- Draft PR에 Human 이해도 답변이 작성됨
- 다른 팀원의 AI 1차 리뷰가 등록됨

동작:

1. 최신 Head와 1차 리뷰를 다시 읽는다.
2. Issue 계약 안의 명확한 지적만 수정한다.
3. QUESTION 또는 정책·API·DB 재결정 항목은 수정하지 않고 Human에게 보고한다.
4. 영향받는 테스트·직접 검증·전체 build를 다시 실행한다.
5. 실제 Diff와 PR 본문·증거를 갱신한다.
6. Commit·Push한다.
7. 최종 Human 검토 상태로 보고하고 중단한다.

## 3. 구현 원칙

- 한 번에 하나의 Issue만 처리한다.
- 최신 Issue 계약을 기준으로 구현한다.
- 존재하지 않는 코드·설정·정책을 추정하지 않는다.
- 기존 패키지 구조와 코드 컨벤션을 유지한다.
- 범위 밖 기능·리팩터링·새 기술을 추가하지 않는다.
- 인증 사용자 ID를 클라이언트 입력값으로 신뢰하지 않는다.
- 상태·금액·수량·권한은 실패 후 결과까지 확인한다.
- 일반 컴파일 오류와 테스트 오류는 범위 안에서 수정한다.
- 정책·구조 재결정이 필요한 오류는 Human에게 보고한다.

## 4. 테스트와 검증

`docs/TEST_CONVENTION.md`를 따른다.

- 한글 설명형 테스트명
- `given → when → then`
- 완료 조건과 실제 실패 위험에 연결된 테스트
- 실제 실행 명령과 환경
- `PASS | FAIL | HOLD | NOT_RUN`
- 통과 건수 또는 실패 로그
- 직접 검증 결과
- 보장 범위와 미검증 범위

실행하지 않은 테스트를 PASS로 기록하지 않는다. 전체 build가 실패하면 성공 또는 완료로 표현하지 않는다.

## 5. 실제 Diff 자체 검토

Commit 전에 다음을 확인한다.

- Issue 범위 밖 변경
- 요청부터 응답·저장까지 실제 흐름
- 입력 검증과 예외 처리
- 권한·소유권
- 트랜잭션·Rollback·부분 성공
- 상태·금액·수량 정합성
- 중복·동시 요청 위험
- 테스트와 완료 조건 연결
- 문서·API·DB 변경 반영
- 비밀정보·개인정보 포함 여부

구현 AI의 자체 검토는 PR 1차 리뷰로 계산하지 않는다.

## 6. 구현 후 Human 질문

최신 Issue·Diff·테스트를 기준으로 담당자가 반드시 이해해야 할 질문만 작성한다.

- 작은 변경: 1~2개
- 일반 기능: 3~5개
- 권한·결제·좌석·동시성: 4~6개

구현 AI는 질문만 작성하고 Human 답변·채점·모범 답안을 작성하지 않는다.

## 7. Draft PR 기록

PR에는 실제 확인한 내용만 기록한다.

- 관련 Issue와 검토 수준
- 주요 변경과 제외 범위
- 코드 읽는 순서
- 완료 조건별 구현·검증 증거
- 실제 테스트·build·직접 검증 결과
- 최신 Commit SHA와 CI 상태
- 미검증 범위와 남은 위험
- 구현 후 Human 질문

실행하지 않은 테스트나 CI를 PASS로 기록하지 않는다.

## 8. 즉시 중단 조건

- 문서·Issue·코드가 충돌함
- Human 답변이 모호하거나 미결정 사항이 남음
- 정책·API·DB·상태·권한·트랜잭션 재결정이 필요함
- 다른 담당자 계약 변경이 필요함
- 새로운 라이브러리나 인프라 결정이 필요함
- 데이터 손실·정합성·보안 위험이 있음
- 핵심 검증 환경이나 권한이 없음
- Issue 범위 밖 변경이 필요함

## 9. 완료 경계

구현 AI는 다음까지만 수행한다.

```text
Issue 정리
→ 구현·테스트
→ Commit·Push·Draft PR
→ 1차 리뷰의 범위 안 지적 수정·재검증·Push
```

Approve와 Merge는 Human이 수행한다.
