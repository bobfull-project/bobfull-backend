# BobFull 구현 AI 실행 가이드

## 1. 목적

이 문서는 Codex·Claude Code 등 구현 에이전트가 승인된 Issue를 코드·테스트·검증·Draft PR로 변환할 때 따르는 실행 기준이다.

구현 AI는 승인된 계약을 구현한다. 정책·설계·리뷰 반영 범위와 Merge는 Human이 결정한다.

## 2. 작업 전 필수 입력

| 확인 대상 | 기준 |
|---|---|
| 프로젝트 정책·버전 범위 | `docs/PROJECT_CONTEXT.md` |
| 대상 기능 계약 | `READY` Issue |
| 도메인 영향 | `docs/DOMAIN_DEPENDENCIES.md` |
| Java·Spring 작성 방식 | `docs/CODE_CONVENTION.md` |
| 테스트 작성·실행·증거 | `docs/TEST_CONVENTION.md` |
| Git·PR 규칙 | `docs/GITHUB_RULES.md` |
| PR 기록 형식 | `.github/pull_request_template.md` |

관련 코드·테스트·문서를 실제로 읽고, 존재하지 않는 구현이나 정책을 추정하지 않는다.

## 3. 시작 조건

다음을 모두 만족할 때만 구현한다.

- Issue가 Human에 의해 `READY`로 승인됨
- 범위·제외 범위·완료 조건이 명확함
- 구현을 막는 미결정 정책이 없음
- 필요한 API·DB·상태 계약이 확정됨
- 관련 담당자 검토가 필요한 경우 완료됨

승인 전에는 코드·테스트·Commit·PR을 만들지 않는다.

다음 상황에서는 즉시 중단한다.

- 확정 문서와 Issue가 충돌함
- 정책·API·DB·트랜잭션 재결정이 필요함
- 다른 담당자 계약을 변경해야 함
- 데이터 손실·정합성·권한·보안 위험이 발견됨
- 범위 밖 기능·리팩터링·기술 도입이 필요함
- 테스트를 삭제하거나 약화해야 함
- 핵심 검증 환경이나 권한이 없음

## 4. 구현 전 계획

코드 수정 전에 최신 코드 기준으로 짧은 계획을 세운다.

- 변경할 파일과 역할
- 요청·데이터 흐름
- 상태 변경과 트랜잭션 경계
- 다른 도메인과의 연결
- 정상·실패·경계 테스트
- 직접 검증 방법
- 현재 확인하지 못한 위험

계획은 Issue 범위 안의 최소 변경이어야 한다. 승인된 구현을 위해 같은 허가를 반복 요청하지 않는다.

## 5. 구현 원칙

- 한 번에 하나의 `READY` Issue만 처리한다.
- Issue 완료 조건을 코드와 검증 근거로 연결한다.
- 기존 구조와 컨벤션을 우선한다.
- 관련 없는 파일을 수정하지 않는다.
- 불필요한 추상화·범용화·리팩터링을 추가하지 않는다.
- 요구사항에 없는 Redis·Kafka·비동기·새 라이브러리를 임의로 도입하지 않는다.
- 인증 사용자 ID를 클라이언트 입력값으로 신뢰하지 않는다.
- 상태·금액·수량은 실패 후 결과까지 일관되게 유지한다.
- 외부 연동이 포함되면 부분 성공과 보상 범위를 명확히 한다.

일반적인 컴파일 오류·테스트 오류는 승인 범위 안에서 수정할 수 있다. 정책이나 구조 재결정이 필요한 오류는 Human에게 보고한다.

## 6. 테스트 작성

`docs/TEST_CONVENTION.md`를 반드시 따른다.

- 테스트 메서드명은 한글 설명형으로 작성하고 단어는 `_`로 구분한다.
- 본문은 `// given`, `// when`, `// then` 순서로 작성한다.
- 하나의 테스트는 하나의 핵심 결과를 검증한다.
- Issue 완료 조건과 실제 실패 위험을 기준으로 필요한 테스트만 작성한다.

검토 대상 후보:

- 정상 흐름
- 입력 오류와 실패 흐름
- 상태 경계
- 권한과 소유권
- 트랜잭션과 Rollback
- 중복 요청과 동시 요청
- 정원·금액·상태 정합성
- 외부 연동 부분 실패

모든 종류의 테스트를 기계적으로 추가하지 않는다. 결제·환불·예약·좌석·상태 변경·권한·동시성은 실패 이후 DB 상태까지 우선 확인한다.

## 7. 테스트 실행과 직접 검증

작성한 테스트와 실행한 테스트는 별개의 사실이다.

PR에는 다음을 기록한다.

- 테스트 클래스와 한글 메서드명
- 테스트 작성 이유
- `given·when·then` 핵심 조건
- 실행 명령과 환경
- `PASS | FAIL | HOLD | NOT_RUN`
- 통과 건수 또는 실패 로그
- 직접 검증 결과
- 보장 범위와 한계
- 최신 검증 대상 Commit SHA
- GitHub Actions 상태와 링크

실행하지 않은 테스트를 `PASS`로 기록하지 않는다.

테스트만으로 증명하기 어려운 경우 실제 관찰 결과를 남긴다.

- HTTP 응답
- DB 상태
- 로그
- 동시 요청 성공·실패 건수
- 캐시·메시지·외부 연동 결과

환경이 없어 실행하지 못한 검증은 `HOLD` 또는 `NOT_RUN`으로 남기고 이유와 위험을 적는다.

## 8. 실제 Diff 자체 검토

Commit 전에 예상 변경이 아니라 실제 Diff를 확인한다.

필수 확인:

- Issue 범위 밖 변경이 없는가
- 요청 진입점부터 저장·응답까지 흐름이 연결되는가
- 검증 순서와 트랜잭션 경계가 적절한가
- 실패 시 부분 데이터가 남지 않는가
- 상태·금액·수량이 일관적인가
- 권한·소유권 검증이 빠지지 않았는가
- 중복·동시 요청 위험을 다뤘는가
- 테스트가 실제 완료 조건을 증명하는가
- 문서·API·DB 변경을 함께 반영했는가
- 비밀정보나 실제 개인정보가 포함되지 않았는가

발견한 일반 오류는 범위 안에서 수정하고 다시 검증한다.

## 9. 코드 리뷰 가이드 작성

Draft PR에는 최신 Diff를 읽을 순서를 작성한다.

```text
요청 진입
→ 핵심 Service
→ Domain·Entity 상태 변경
→ Repository·외부 연동
→ 응답·이벤트
→ 테스트
```

각 단계에 실제 파일·클래스·메서드, 역할과 확인할 핵심을 적는다. 변경 파일을 단순 나열하지 않는다.

리뷰어 확인 포인트에는 이번 PR에서 사고 가능성이 높은 부분만 적는다.

- 집중해서 볼 코드·흐름
- 실패 가능성이 큰 지점
- 테스트에서 확인할 증거
- Human 판단이 필요한 설계

## 10. 구현 후 이해도 질문

최신 Issue·Diff·테스트 결과를 기준으로 담당자가 반드시 이해해야 할 내용만 질문한다.

- 작은 변경: 1~2개
- 일반 기능: 3~5개
- 결제·좌석·권한·동시성 등 중요 기능: 4~6개

질문 주제:

- 기능 목적과 요청·데이터 흐름
- 클래스별 역할
- 상태 변경과 트랜잭션
- 실패 시 남는 결과
- 권한·소유권·동시성
- 테스트가 보장하는 범위와 한계
- 중요한 선택 이유

신입 백엔드 기술면접 수준을 유지한다. 구현 AI는 질문만 작성하며 답변·채점·모범 답안을 제공하지 않는다.

## 11. Commit·Push·Draft PR

다음을 완료한 뒤 Git 작업을 수행한다.

- 구현과 필요한 문서 반영
- 테스트·직접 검증
- 실제 Diff 자체 검토
- 미검증 범위 정리
- 이해도 질문 작성

Git 작업은 `docs/GITHUB_RULES.md`를 따른다.

- 기능 브랜치에서 작업한다.
- 관련 변경만 Commit한다.
- Commit·Push 후 Draft PR을 작성한다.
- PR 대상 브랜치는 GitHub Rules를 따른다.
- Merge하지 않는다.

PR 본문에는 Issue 연결, 주요 변경, 제외 범위, 코드 읽는 순서, 리뷰어 확인 포인트, 테스트·검증 증거, 미검증 위험과 이해도 질문을 기록한다.

## 12. 리뷰 수정

ChatGPT 또는 Human 리뷰가 등록되어도 구현 AI가 임의로 전부 반영하지 않는다.

1. Human이 반영할 내용을 결정한다.
2. 구현 AI는 승인된 항목만 수정한다.
3. 영향받은 테스트와 검증을 다시 실행한다.
4. PR 설명·증거·최신 Commit을 갱신한다.
5. ChatGPT 최신 Head 재검토를 요청한다.

리뷰가 Issue 범위를 확장하거나 정책·API·DB 재결정을 요구하면 수정하지 않고 Human 판단을 요청한다.

## 13. 완료 인계 체크리스트

- [ ] `READY` Issue 범위만 구현했다.
- [ ] 확정 정책과 도메인 영향을 확인했다.
- [ ] 불필요한 기술·리팩터링을 추가하지 않았다.
- [ ] 테스트를 한글명과 given-when-then으로 작성했다.
- [ ] 실제 실행 명령·결과·증거를 기록했다.
- [ ] 미실행 테스트를 `PASS`로 기록하지 않았다.
- [ ] 실패 후 상태와 미검증 위험을 확인했다.
- [ ] 실제 Diff 기준 코드 읽는 순서를 작성했다.
- [ ] 담당자 이해도 질문을 작성했다.
- [ ] 최신 검증 Commit과 CI 상태를 기록했다.
- [ ] Draft PR까지만 작성했고 Merge하지 않았다.
